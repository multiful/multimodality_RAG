# 리서치 PDF 이미지 파이프라인 고도화 멘토링 보고서

작성 기준일: 2026-07-22

## 1. 한 줄 요약

한국 증권사 리서치 PDF의 차트·이미지를 대상으로, 기존의 `MinerU → Qwen3-VL-8B 전량 처리` 구조에 경량 그림 분류기와 DePlot을 실험적으로 결합했다. 그 결과 단일 모델로 분류·한글 인식·수치 추출을 모두 해결하기 어렵다는 점을 확인했고, `분류기=저장 선별`, `Qwen3-VL=한글 엔티티·요약`, `DePlot=단일계열 수치 추출`로 역할을 분리하는 하이브리드 방향을 도출했다.

> 주의: 모델 가중치를 파인튜닝한 프로젝트가 아니라, 사전학습 모델의 프롬프트·추론 설정·라우팅·후처리·평가 구조를 고도화한 프로젝트다.

## 2. 프로젝트 목표와 범위

- 목표: 네이버 금융의 한국 증권사 리서치 PDF에서 투자 분석에 유용한 차트와 이미지를 선별하고, OCR·요약·기업명·티커·지표명을 구조화해 이후 RAG 검색에서 사용할 수 있게 한다.
- 현재 실측 데이터: 하나증권 산업분석 PDF 20건.
- 담당 범위: 이미지 파트. MinerU의 `chart`와 `image` 크롭을 처리하며 `table`은 별도 파트로 인계한다.
- 실행 환경: Windows, RTX 5080 16GB, 로컬 Ollama, CUDA 격리 가상환경.

## 3. 사용·비교한 모델

| 구분 | 모델/도구 | 역할 | 현재 판단 |
|---|---|---|---|
| 문서 파싱 | MinerU 3.4.x, DocLayout-YOLO | PDF의 본문·차트·이미지·표 블록 탐지 및 크롭 | 20건 파싱 완료. 차트 내부 수치는 읽지 못함 |
| 베이스라인 VLM | Qwen3-VL-8B, Ollama | 이미지 유형, useful 여부, OCR, 요약, 엔티티 추출 | 한글 라벨·범례와 엔티티에 가장 강하지만 느리고 수치 구조화가 약함 |
| 경량 분류기 | DocumentFigureClassifier-v2.5, EfficientNet-B0 4.08M | 로고·사진·아이콘 등 명백한 junk 선별 | 약 10~20ms/장. 저장 정밀도와 비용에 효과적 |
| 차트 수치 모델 | google/deplot, Pix2Struct | 차트를 표 형태의 시계열 수치로 변환 | 단일계열 수치는 강하지만 한글 범례가 깨짐 |
| 비교 모델 | ChartGemma, PaliGemma-3B | 차트 질의응답 및 한글 범례 인식 비교 | 한글 범례는 읽지만 수치 답변이 부정확해 미채택 |
| 검토 후 보류 | TinyChart | 더 강한 단일 차트 모델 후보 | Windows/Blackwell 환경과 모델 코드 의존성, 12.8GB 크기로 보류 |
| 검색 후보 | BGE-M3 | 이미지 설명 텍스트 임베딩 | 유틸리티만 존재하며 현재 실제 색인·검색에는 미연결 |

## 4. 기존 구조의 문제

1. 모든 이미지에 Qwen3-VL을 호출해 사진·로고에도 OCR 비용이 발생했다.
2. Qwen3-VL이 로고 그리드, 제품 사진, 조감도를 useful로 판정하는 거짓양성 사례가 있었다.
3. Qwen3-VL의 `ocr_text`는 차트 축 눈금과 범례는 읽지만 실제 시계열 값을 구조화하지 못했다.
4. 하나의 VLM 호출에 분류·OCR·요약·엔티티 추출을 묶어, 유용성 판단 전에 비싼 처리가 이미 수행됐다.
5. confidence가 품질을 충분히 구분하지 못했다. 현재 판정 172건 중 132건이 정확히 0.95이고 24건이 1.0으로, 보정되지 않은 자기보고 점수에 가깝다.
6. 저장 목적의 useful/junk 판단과 검색 목적의 엔티티 보존을 같은 게이트로 처리하면, 로고 그리드 같은 이미지의 기업 엔티티를 잃게 된다.

## 5. 적용한 고도화

### 파이프라인 안정성과 비용

- `content_hash + prompt_ver + model` 기반 VLM 캐시를 도입했다.
- dHash 해밍거리 0은 판정을 복사하고, 1~6은 유사 표시만 한 뒤 다시 판정하도록 설계했다. 시계열 차트의 잘못된 판정 복사를 방지하기 위한 선택이다.
- confidence 0.6 미만은 `review_queue`로 자동 분리했다.
- table 블록은 VLM을 호출하지 않고 `handoff_tables.jsonl`로 인계했다.
- 결정적 `image_id`, 완료 건 스킵, 개별 실패 격리, Ollama 5연속 실패 시 중단으로 resume 구조를 만들었다.
- 원본 탈락 이미지를 삭제하지 않고 보관해 재검수와 튜닝 근거로 남겼다.

### 모델 라우팅

- `--route clf` 옵션으로 경량 그림 분류기 게이트를 추가했다.
- 고신뢰 로고·사진·아이콘만 선컷하고, `full_page_image`처럼 데이터 포함 가능성이 있는 유형은 VLM으로 보낸다.
- DePlot을 VLM 대체 모델로 사용하지 않고 단일계열 차트의 수치 골격을 추출하는 보조 모델로 재정의했다.
- DePlot의 깨진 문자열과 축 라벨이 엔티티 합집합을 오염시키는 문제에 `clean_deplot_entities()` 필터를 적용했다.

### 추론 안정화

- 큰 차트에서 Ollama 응답이 비거나 JSON 파싱이 실패하는 문제를 줄이기 위해 이미지 장변을 1400px로 제한했다.
- VLM 컨텍스트를 16,384로 확대했다.
- Qwen3-VL의 thinking이 출력 예산을 소모하는 문제를 피하려고 `think=false`로 설정하고 `num_predict` 상한을 두지 않았다.
- JSON 파싱 실패는 confidence 0.3의 보류 카드로 저장해 배치를 계속 진행하도록 했다.

## 6. 주요 실험 결과

### 엔티티 추출 A/B

표본 12장, 골든 엔티티 56개 기준이다.

| 구성 | 총시간 | Precision | Recall | F1 | 해석 |
|---|---:|---:|---:|---:|---|
| V1 전량 Qwen3-VL | 326.0초 | 92.7% | 91.1% | 91.9% | 느리지만 엔티티 회수 최고 |
| V2 분류기 + Qwen3-VL | 221.7초 | 100.0% | 62.5% | 76.9% | 32% 빨라졌지만 junk 내부 엔티티 손실 |
| V3 분류기 + DePlot | 28.5초 | 7.4% | 3.6% | 4.8% | 한글 범례 오독으로 엔티티 추출 붕괴 |
| V3' DePlot + VLM 원시 합집합 | 250.0초 | 58.3% | 62.5% | 60.3% | 깨진 문자열과 축 라벨이 거짓양성 유발 |
| V4 필터 하이브리드 | 250.0초 | 100.0% | 62.5% | 76.9% | 필터로 정밀도 회복, recall 문제는 잔존 |

핵심 해석:

- 엔티티 품질만 보면 전량 Qwen3-VL이 현재 최고다.
- 분류기 도입은 속도와 저장 정밀도에는 유리하지만, 엔티티 추출까지 중단하면 recall이 하락한다.
- DePlot은 엔티티 모델이 아니라 수치 추출 모델로 제한해야 한다.
- V4의 Precision 100%는 어려운 junk 이미지를 처리하지 않은 영향이 크므로, 완전한 품질 향상으로 과장하면 안 된다.

### 현재 파이프라인 산출물

- 원본 PDF: 20건 수집, 전부 하나증권 산업분석.
- MinerU 파싱: 20건 완료, 실패 0건.
- 이미지 파이프라인: 현재 5개 문서 처리, 고유 카드 233건.
- 블록: chart 170건, table 61건, image 2건.
- 상태: useful 171건, table handoff 61건, VLM discard 1건.
- VLM 판정 대상 172건: 실제 호출 148건, 캐시 24건, 완전중복 복사 0건.
- 현재 누적 캐시 절감률 14.0%. 동일 2개 문서 재실행에서는 VLM 0회, 캐시 100% 적중을 확인했다.
- 유사 이미지 표시 13건, 정확 중복 0건.
- review queue 3건. 모두 JSON 파싱 오류 fallback이며 confidence 0.3이다.
- VLM 속도: p50 22.7초, p95 46.8초, 최대 141.3초.
- 파싱 속도: 문서당 p50 약 69.5초, 최대 369.2초.
- table 61건의 VLM 호출은 0건으로 인계 원칙을 지켰다.

## 8. 현재 부족한 점

1. **모델 파인튜닝은 하지 않았다.** 현재 성과는 라우팅·후처리·프롬프트·인프라 고도화다.
2. **전체 이미지 배치가 미완료다.** 20건 파싱은 끝났지만 이미지 카드는 5개 문서까지만 생성됐다.
3. **핵심 useful 판정 정확도가 아직 없다.** `eval/image_labels.csv`가 없어 accuracy, precision, recall과 로고 FP율을 계산하지 못했다.
4. **실험 표본이 작고 편중됐다.** 엔티티 평가는 반도체 중심 12장이며, 로고 그리드 한 장이 골든 56개 중 16개를 차지한다.
5. **골든셋이 단일 LLM 판독이다.** 사람 또는 두 번째 판독자의 교차검증이 필요하다.
6. **하이브리드 결과가 메인 파이프라인에 아직 통합되지 않았다.** DePlot과 V4 필터는 벤치·평가 코드에 있고 `s2_image_pipeline.py` 기본 실행은 여전히 Qwen3-VL 전량 처리다.
7. **분류기 라우팅도 기본값이 아니다.** `run_full20.ps1`은 `--route clf` 없이 실행된다.
8. **설계와 구현의 스키마가 다르다.** 마스터 설계에는 `chart_data`가 있지만 실제 `PROMPT_V2`와 저장 매핑에는 없고, 현재 카드 172건 모두 `chart_data`가 없다.
9. **confidence가 보정되지 않았다.** 0.95에 과도하게 몰려 있어 0.6 임계값의 품질 의미가 약하다.
10. **JSONL 중복 append가 있다.** 이미지 카드는 275행/고유 233건, table handoff는 79행/고유 61건이다. 최신 상태 조회는 되지만 리포트 지표와 운영 데이터가 혼동될 수 있다.
11. **Supabase 운영 연결이 없다.** 환경변수 미설정으로 DB·Storage 저장이 모두 생략됐고 `storage_path`는 전부 비어 있다. `pipeline_status`, DB 스키마 적용 스크립트도 미구현이다.
12. **BGE-M3 검색과 최종 RAG 평가가 없다.** `embed_text` 문자열은 171건 생성됐지만 실제 임베딩·색인·질의 Recall@K는 측정하지 않았다.
13. **수치 검증 체계가 없다.** DePlot 값이 축 범위·합계·단위·원문 캡션과 일치하는지 자동 검증하지 않는다.
14. **현재 useful 비율이 99.4%로 지나치게 높다.** VLM 플래그와 유용 유형을 OR로 결합하는 로직 때문에 유형이 차트면 false 판정이 뒤집힐 수 있어 추가 검증이 필요하다.

6. VLM confidence 대신 OCR·구조·문맥 일치도를 분리해 검증하는 방식이 타당한가?
7. `chart_data`를 VLM 프롬프트로 직접 생성하는 방식과 DePlot 구조를 후처리하는 방식 중 어떤 것을 우선 실험해야 하는가?
8. 전체 RAG 품질을 증명하려면 이미지 단독 평가 이후 어떤 질의셋과 Recall@K·citation accuracy를 구축해야 하는가?
9. 현재 로컬 JSONL 중심 구조를 유지하면서 Supabase를 운영 저장소로 전환할 때 필요한 최소 manifest와 재처리 정책은 무엇인가?

## 10. 다음 개선 계획

### 1순위: 평가 완성

- 50~100장 수동 라벨링으로 useful 판정 accuracy, precision, recall, 로고 FP를 산출한다.
- 엔티티 골든셋을 최소 100~200장으로 확대하고 사람 교차검증을 추가한다.
- 전체 20개 문서 이미지 배치를 resume로 완료한다.

### 2순위: 메인 파이프라인 통합

- 기본 실행을 `분류기 트리아지 → Qwen3-VL 엔티티 → 단일계열만 DePlot 수치`로 변경한다.
- 저장 게이트와 엔티티 게이트를 분리한다.
- V4 필터를 평가 코드가 아닌 실제 산출물 생성 단계에 적용한다.

### 3순위: 정보 구조 개선

- `PROMPT_V3`에 `metric`, `period`, `unit`, `actual_or_forecast`, `series`, `points`를 추가한다.
- classification, OCR, structure confidence를 분리하고 자동 검증 결과를 저장한다.
- JSONL은 감사 로그로 유지하고 고유 최신 상태의 compact snapshot을 별도로 만든다.

### 4순위: 검색·운영 검증

- BGE-M3 색인과 실제 금융 질문 기반 Recall@5를 측정한다.
- Supabase `image_cards`, Storage, `pipeline_status`를 연결한다.
- 문서·모델·프롬프트 버전별 처리 이력을 manifest로 관리한다.

## 11. 3분 발표용 요약

“저는 한국 증권사 리서치 PDF의 이미지 파이프라인을 고도화했습니다. 베이스라인은 MinerU가 차트와 이미지를 크롭하고 Qwen3-VL-8B가 전부 판독하는 구조였습니다. 문제는 사진과 로고에도 장당 수십 초의 VLM 비용이 발생하고, Qwen은 한글 범례는 잘 읽지만 차트 실제 수치를 구조화하지 못한다는 점이었습니다.

그래서 EfficientNet-B0 기반 그림 분류기를 앞단에 넣어 명백한 junk를 약 15ms에 선별하고, 차트 수치 전문 모델 DePlot을 비교했습니다. 12장, 골든 엔티티 56개 실험에서 전량 Qwen은 F1 91.9%, 분류기 추가는 시간이 326초에서 222초로 줄었지만 recall이 62.5%로 하락했습니다. DePlot 단독 엔티티 추출은 한글 범례가 깨져 F1 4.8%로 실패했고, VLM과 단순 결합해도 노이즈 때문에 Precision이 58.3%까지 떨어졌습니다. 이후 깨진 문자열과 축 라벨 필터를 적용해 Precision 100%, F1 76.9%까지 회복했습니다.

결론은 단일 모델 교체가 아니라 역할 분리입니다. 분류기는 저장 선별, Qwen3-VL은 한글 엔티티와 요약, DePlot은 단일계열 수치 추출에 사용해야 합니다. 다만 전체 20건 중 이미지 처리는 현재 5개 문서까지만 완료됐고, useful 판정 수동 라벨과 메인 파이프라인의 하이브리드 통합은 아직 남아 있습니다. 멘토링에서는 저장 게이트와 엔티티 게이트 분리 방식, 평가셋 확장 기준, 차트 수치 정답셋 설계를 조언받고 싶습니다.”
