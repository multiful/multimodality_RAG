# 고도화 파이프라인 개선 요약 (baseline → enhanced_v5)

> 4축(baseline / docling / MinerU / 고도화) PDF 파싱 비교에서 **고도화 파이프라인을 v1→v5로 개선**하고 **하나증권 15개 보고서 A/B로 검증**한 과정 요약.
> 전체 상세는 [실험_4축_비교_스마트폰.md](실험_4축_비교_스마트폰.md) §0~17.

---

## TL;DR

**최종 채택 = enhanced_v5.** 15문서 A/B에서 **엔티티 recall 최고(0.835)**, baseline을 **통계적으로 압도(p<0.0001)**, docling/MinerU 대비 **~10× 빠름**, 캡션 100%, 페이지분류·라우팅 보유.

| 파서 | 엔티티 recall (15문서) | 파싱 평균 | vs enhanced_v5 |
|---|---:|---:|---|
| **enhanced_v5** | **0.835** | **8.5s** | — |
| docling | 0.788 | 81.3s | +18 (p=0.11) |
| MinerU | 0.786 | 97.5s | +19 (p=0.096) |
| baseline | 0.672 | 0.07s | **+63 (p<0.0001)** |

핵심: **소표본(단일/3/5문서)에선 우열이 엎치락뒤치락했지만, 15문서로 키우자 enhanced_v5 우위가 확정**됐다. 표가 많은 문서일수록 구조화(고도화)의 값어치가 커지는데, 큰 표본이 이런 문서를 담으며 진짜 신호가 드러났다.

## 고도화 계보 — 무엇을 어떻게 고쳤나 (v1 → v5)

각 단계는 **"증상 → 진단(present/absent 테스트) → 근본원인 → 수정 → 검증"** 루프로 진행. 대부분 *정보 파괴를 멈추거나 흘린 걸 회수*하는 방식이라 **품질↑와 속도유지가 동시에** 가능했다.

| 버전 | 문제 | 수정 | 결과 |
|---|---|---|---|
| **baseline→v1** | 구조·분류·라우팅 없음 | YOLO 페이지분류 + 라우팅 + TATR canonical + 계층청킹 도입 | 구조/분류 획득. 단 **표 파편·캡션 과삭제** 잔존 |
| **v1→v2** | ① wide-form 표 canonical collapse로 회사↔값 끊김 ② 캡션 삭제 | ① 파이프라인 row parser 행단위 재선형화 ② 원문 캡션 회수 | 표 정상행(`\| 삼성전자 \| … \| 7.6 \|`), 캡션 87→**93/93**, **파싱 47.6→30.1s** |
| **v2→v3** | ① 본문 엔티티 유실(계층청킹) ② 표 값 유실(TATR 컬럼 truncation) ③ p31 오라우팅 | ① ② 완전성 레이어(정제 원문을 검색 recall용 청크로 회수) ③ 컴플라이언스 보일러플레이트 하드 취소 | **커버리지 1.0**, 검색 hit@5 0.90→**0.967**, 라우팅 오탐 0 |
| **v3→v4** | 구조화 표 레코드가 여전히 우측열 누락(DB 경로 결함) | 표 추출을 **pdfplumber text-strategy**로 교체(TATR 제거) | 구조화 DB 완전(479.0 포함), **파싱 30.8→21.0s** … 단 **단일문서 과적합** |
| **v4 회귀 발각** | **5문서 A/B**: text-strategy가 불규칙 표(학회일정 등)에서 셀 임의분할 → **회사명 쪼갬**(에이비엘바이오→에이비엘\|바이오) | — | 제약/철강 엔티티 recall 하락 |
| **v4→v5 (FINAL)** | text-strategy가 정형표엔 완벽하나 불규칙표엔 취약 | **하이브리드 게이트**: `(열 일관성≥0.5 & 최빈열≥2) OR (수치비율≥0.55)` → text-strategy, 아니면 **word-clustering**(이름 보존) | **다문서 검증서 v4 회귀 해소**, 파싱 8.5s로 더 빠름 |

### 표본 확대가 뒤집은 것 (정직한 기록)
- **단일문서(SmartPhone)**: v3/v4가 커버리지·검색에서 최고로 보였음.
- **3문서**: enhanced가 엔티티 recall 최고(오해의 소지 — 표본 특이적).
- **5문서**: v4가 불규칙표 회귀로 오히려 근소 최하 → **단일문서 과적합 발각**.
- **15문서**: v5가 엔티티 최고 + baseline 유의 압도로 **확정**.
- **교훈: "고도화"는 다문서 검증을 통과할 때만 고도화다.** v4는 단일문서 지표에 과적합했고, v5는 다문서 검증을 개선 조건으로 삼아 그 함정을 피했다.

## 최종 15문서 A/B (가장 단단한 결론)

**대상**: 하나증권 산업분석 15종(건설·제약·프리뷰·리테일·철강·항공·마트·태양광·은행·식품·납기·관광·건설2·반도체·실적).
**엔티티 정답셋**: consensus(2개+ 파서 합의) 387개. **채점**: 파이프라인 기본 추출기(gpt-4o-mini) + pooled McNemar.

| 지표 | 결과 |
|---|---|
| **엔티티 recall** | enhanced_v5 **0.835 최고** (docling 0.788 / MinerU 0.786 / baseline 0.672) |
| **유의성** | v5 ≫ baseline **p<0.0001**; v5 vs docling/MinerU **p≈0.10**(근소 우세, 유의 근접) |
| **파싱 지연** | enhanced_v5 **8.5s** = docling(81s)·MinerU(98s) 대비 **9.5~11.4× 빠름** |
| **캡션 보존** | 4파서 ~100% (docling 98.7%) |
| **페이지분류·라우팅** | enhanced_v5 전용(15문서서 라우팅 오버라이드 정상 작동) |

**정직한 단서**: (a) doc15 실적(146개, pool의 38%)이 비중 큼, (b) vs docling/MinerU는 p≈0.10로 유의화엔 20~30문서 필요, (c) consensus-GT는 사람 라벨 아닌 상대 지표.

## 결론

- **enhanced_v5 = 품질(엔티티 최고·baseline 유의 우세) + 속도(~10×) + 구조·분류·라우팅**을 동시에 달성한 최선.
- **baseline**은 깨끗한 PDF 검색엔 값싸게 경쟁하나 **엔티티(=구조화 DB 저장)에서 유의하게 최하** → DB 파이프라인 부적합.
- **docling/MinerU**는 품질 대등하나 **~10× 느림** — 전 페이지 무거운 파싱 vs 고도화의 라우팅.
