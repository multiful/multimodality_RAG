# 전체 파이프라인 vs 베이스라인 비교 — LG CNS, Construct (사용자가 최초에 보낸 2개 레퍼런스 PDF)

사용자 요청: "이제 전체 파이프라인 내가 보낸 2피디에프에 대해서 베이스라인과 현재파이프라인
비교해". `page_classification`(공유 YOLO) -> `text_processing.process_pdf()` -> `table_processing`
(Adaptive Router+TATR+canonical field 매칭)을 실제로 연결해서 두 PDF에 돌리고, "라우팅/정제/
청킹/canonical 매칭이 전혀 없는 순수 추출"과 비교했다. 스크립트: `pdf_pipeline/
compare_baseline_vs_pipeline.py`. (이미지 파트는 팀원 모듈을 가져다 붙일 예정이라 이번 비교에서
제외 — 사용자 확인.)

## 베이스라인 정의

- **텍스트**: PyMuPDF `page.get_text()` 그대로(PUA 제거/헤더푸터 제거/구두점 정규화조차 없음)
- **표**: 공유 YOLO로 찾은 Table bbox에 pdfplumber `extract_table()`만 적용(Adaptive Router의
  SIMPLE/COMPLEX 판단도, TATR도, canonical field 매칭도 없음) — 병합 셀이 많은 표에서 대개
  1행짜리 쓰레기가 나온다는 게 핵심이라 canonical 매칭 자체를 시도하지 않음
  (`benchmark_construct_validation.py` [23]와 동일 관례)

## 결과

### 텍스트 (골든셋 recall, 육안으로 직접 작성한 서술형 콘텐츠 기준)

| PDF | baseline recall | current recall | 비고 |
|---|---|---|---|
| LGCNS | 100.0%(22/22), 41.7ms | 100.0%(22/22), 5.16s, 17청크 | hard 페이지 0개 |
| Construct | 100.0%(47/47), 47.7ms | 100.0%(47/47), 1.15s, 20청크 | hard 페이지 0개 |

**재측정 중 발견한 측정 아티팩트**: 첫 실행에서 Construct recall이 93.6%(44/47)로 나와 조사했다.
Construct page5(뉴스 헤드라인 3건)가 원인이었는데, 실제로는 **정보 손실이 전혀 없었다** —
`normalize_punctuation()`이 "…"(U+2026)를 "..."(ASCII 마침표 3개)로 바꾸는 게 의도된 동작인데,
골든셋은 렌더링된 페이지를 육안으로 옮겨 적어 원본 문자 "…"를 그대로 쓰고 있어서 recall 채점
스크립트가 이 차이를 문자열 불일치로 잘못 셌던 것. 골든셋 비교 시 ellipsis도 같이 정규화하도록
고치자 3건 모두 정상 매칭되어 **Construct도 100%(47/47)**. 즉 baseline 대비 current가 텍스트
정보를 하나도 잃지 않으면서 정제+청킹+구조 메타데이터(section_path)를 추가로 얻는다.

### 표 (같은 표 위치 기준, canonical field는 `sector_schema.yaml` 스키마 대비)

| PDF | baseline 행수 | baseline 0행 표 | current 행수 | canonical 매칭 | hit_rate | 재무항목 필터 |
|---|---|---|---|---|---|---|
| LGCNS | 30행(14개 표 중) | 4개 | **227행** | 27개 | 11.9% | 72행 |
| Construct | 47행(15개 표 중) | 2개 | **215행** | 9개 | 4.2% | 0행 |

**행 추출량**: LGCNS 30→227행(7.6배), Construct 47→215행(4.6배) — Adaptive Router가 SIMPLE(pdfplumber
그대로)/COMPLEX(TATR grid)를 표마다 나눠 판단해, 병합 셀 때문에 pdfplumber 단독으로는 통째로
0~1행만 나오던 표들(baseline에서 0행 표 LGCNS 4개, Construct 2개)까지 제대로 구조화한 결과.

**canonical field hit_rate가 낮은 이유(11.9%/4.2%)**: 이건 결함이 아니라 설계상 당연한 결과다.
`[22]`에서 "순수 재무항목(매출액/영업이익 등)은 이미 DB에 있으니 행 단위로 걸러내고 세그먼트
정보만 남긴다"고 결정했기 때문에, canonical 매칭 시도 대상 자체가 "이미 스키마가 커버하는
쉬운 항목은 다 빠진 나머지"다. 실제로 LGCNS의 매칭 27건은 전부 page3의 wide-form 표(헤더=필드명)
하나에서 나왔고, 나머지 12개 표(대부분 finance 표에서 재무항목 필터를 거치고 남은 세그먼트별
매출 등)는 canonical 매칭이 0건 — `sector_schema.yaml`의 alias가 아직 이 세그먼트 레벨 항목까지는
커버하지 못한다는 뜻. Construct(건설업)는 9건만 매칭(page3 wide-form 표)되고 나머지 14개 표
(청약/분양/수주공시류)는 대부분 0건 — 건설업 sector_schema 커버리지가 상대적으로 더 얕다는 신호.

## 종합 결론

| 항목 | 판정 |
|---|---|
| **텍스트 파이프라인** | **정보 손실 없음(100%/100%)** — 정제+청킹+구조 메타데이터를 공짜로 얹음. 인덱싱 진행에 결격사유 없음. |
| **표 파이프라인 — 행 추출** | **베이스라인 대비 4.6~7.6배 개선** — 병합 셀/복잡한 표 구조를 Adaptive Router+TATR이 사람이 읽는 것과 비슷한 수준으로 복원. |
| **표 파이프라인 — canonical 매칭** | **낮음(11.9%/4.2%), 하지만 설계 의도대로**. rule-based 매칭은 "이미 아는 필드"만 잡고, 나머지(세그먼트/건설업 청약 정보 등 스키마 밖 정보)는 원문 그대로 보존됨(버려지지 않음 — records에 canonical_field=None으로 남아있음). |
| **실제 남은 갭** | `sector_schema.yaml`의 필드 커버리지가 아직 두 PDF의 표 내용을 다 담지 못함 — 매칭 안 된 나머지 rows/segment 정보를 어떻게 쓸지가 다음 결정 포인트. |
| **이 갭을 메우는 담당** | `[25]` 구조화 출력의 `unmapped_fields_summary` 필드가 정확히 이 역할(매칭 안 된 행이 뭔지 요약)로 설계되어 있음 — canonical 매칭 확장 대신 구조화 출력으로 보완하는 것도 선택지. |
| **인덱싱 진행 가능 여부** | **텍스트는 바로 가능.** 표는 canonical 매칭된 27+9건은 구조화 데이터로, 나머지는 raw_label/cells 그대로 원문 보존된 상태로 인덱싱 가능(정보 유실 없음) — 다만 "매칭 안 된 표 정보를 검색에서 어떻게 노출할지"는 인덱싱 스크립트 설계 시 결정 필요. |

## 추가: K-Wave(73페이지, 표 104개) 분석 — "실적이 메타데이터로 잘 나오는가"

사용자 요청: K-Wave 산업 리포트(다수 엔터테인먼트 기업 커버, `reference/KWave/
20260721_industry_65157000.pdf`)도 분석하되, 특히 "실적"(재무 성과)이 메타데이터로 잘
드러나는지 확인. 스크립트: `pdf_pipeline/analyze_kwave.py`. 이 PDF는 텍스트 골든셋이 없어
recall은 측정 못 함(정직하게 범위 밖으로 남김) — 대신 처리량/구조화 출력 품질에 집중.

### 텍스트

135청크 추출(11.0초), 그런데 **73페이지 중 33페이지(45%)가 "hard"로 분류**됨 — LGCNS(0/6),
Construct(0/10)와 뚜렷이 다름. 다수 기업을 표+차트 위주로 다루는 산업 리포트 특성상 컬럼
중첩/그림 밀도가 높은 페이지가 많다는 뜻으로 추정(원인 심층 분석은 이번 범위 밖) — 실제
인덱싱 시 이 33페이지는 우리 파이프라인이 아니라 MinerU 등 외부 리더로 가야 함을 뜻하므로,
"73페이지 전부 우리 파이프라인이 커버한다"고 가정하면 안 됨.

### 표 — baseline vs current

| 항목 | baseline | current |
|---|---|---|
| 총 행수 | 796행 | **2045행**(2.6배) |
| 0행 표 | 15개(104개 중) | - |
| canonical 매칭 | - | 13개(hit_rate **0.64%**) |
| 재무항목 필터 | - | 809행 |

hit_rate 0.64%는 LGCNS(11.9%)/Construct(4.2%)보다도 훨씬 낮다 — 이 리포트가 다수 기업(하이브,
에스엠, 코스맥스엔비티, 노바렉스, 스튜디오드래곤 등)의 손익계산서/재무상태표/현금흐름표를
반복적으로 담고 있어 표 개수 자체는 많지만, 매칭에 성공한 13건은 전부 엔터 섹터 특화
세그먼트 필드(`content_revenue`, `concert_revenue`, `ip_sales`, `fanclub_revenue` 등)였다 —
스키마가 의도한 대로 정확히 작동은 하지만, 커버리지가 이 정도 규모의 다중기업 리포트 전체를
담기엔 아직 얕다는 뜻.

### 구조화 출력 — "실적"이 메타데이터로 잘 나오는가 (finance 분류 표 55개 대상)

**결론: 매우 잘 나온다.** canonical 매칭이 0.64%로 사실상 못 잡는 정보를, 구조화 출력의
`notable_finding`이 회사별로 정확하고 구체적으로 요약해냄. 실제 출력 샘플(전체는
`result_kwave_analysis.json` 참고):

- 하이브: "영업이익률이 4.3%에서 시작해 최저 -5.8%로 하락, 이후 회복세"
- 에스엠: "순이익의 YoY 변화가 1931.3%로 급증"
- 노바렉스: "순이익 증가율이 257.2%로 급증"
- 스튜디오드래곤: "영업이익률이 4.1%에서 9.8%로 증가하며 흑전 유지"
- 코스맥스엔비티: "2025년 전체 매출액이 전년 대비 급감할 것으로 예상됨"

`table_title`도 회사명이 표에 명시 안 돼 있어도("Financial Data Summary" 같은 표에서도) 문맥상
회사명을 정확히 붙여 뽑아냄(예: "하이브 실적 테이블", "에스엠 실적 테이블"). **이게 이번
분석에서 가장 중요한 발견** — canonical field 매칭(rule-based)이 스키마 커버리지 한계로
못 잡는 정보를, 구조화 출력(LLM)이 정확히 그 빈틈을 메운다는 게 실측으로 확인됨.

**비용/지연**: finance 표 55개에 115.6초(표당 평균 2.10초) — 앞서 측정한 LGCNS/Construct의
표당 ~1.9초와 일관됨. 104개 표 전체(finance+contract_or_general 다) + 135청크 텍스트까지 다
켜면 이 문서 하나에 구조화 출력만으로 수 분이 걸릴 것으로 예상 — 앞서 확인한 "직렬 호출이라
느림" 병목이 큰 문서일수록 더 크게 체감된다는 뜻. 병렬화가 이런 대형 문서에서는 사실상 필수.

## 참고: 재현 방법

`python3 pdf_pipeline/compare_baseline_vs_pipeline.py` — 결과는
`pdf_pipeline/result_baseline_vs_pipeline_comparison.json`에 저장. `text_processing/text_normalization.py`와
`table_processing/text_normalization.py`가 이름은 같지만 다른 함수 세트를 가진 별개 파일이라,
같은 프로세스에서 두 모듈을 동시에 import하면 `sys.modules` 캐시 충돌이 나는 것을 발견 —
스크립트 안에서 `importlib`로 격리해 우회했다(각 모듈 파일 자체는 안 건드림). 나중에 실제
인덱싱 스크립트를 짤 때도 같은 문제를 만나게 되니, 이 우회 패턴을 재사용하거나 — 더 근본적으로는
— 두 파일 중 하나 이름을 바꾸는 걸 고려할 것.
