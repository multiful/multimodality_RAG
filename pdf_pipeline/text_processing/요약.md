# 2-4. Text Processing Pipeline

## 측정에 사용한 데이터셋

`20260721_company_279243000.pdf`(교보증권 LG CNS(064400) 기업분석 리포트, 6페이지),
`20260721_industry_362851000.pdf`(하나증권 건설 Weekly 산업 리포트, 10페이지) — 섹터/형식이
다른 두 문서로 처음부터 교차 검증. **범위**: page_classification에서 has_text로 라우팅된 페이지의
서술형 본문을 추출·정제·청킹하는 것까지. 표는 table_processing, 이미지는 다른 팀원 파트 소관 —
실측 결과 두 문서 모두 서술형 텍스트가 있는 페이지는 전체의 20~30%뿐(나머지는 표/차트).

## 베이스라인: PyMuPDF whole_page 추출

**방식**: 페이지 전체를 PyMuPDF `get_text()`로 통째로 추출(표/차트 영역 구분 없음, YOLO 필터링 없음).

| 방식 | Recall(골든셋 5페이지) | 순도 | 지연 |
|---|---|---|---|
| **whole_page(채택)** | **100%(5/5)** | 67.3% | **9.5ms/페이지** |
| yolo_filtered(YOLO Text류 박스만 크롭) | 86.6% | 76.1% | 264.3ms |
| whole_page_masked(표/차트 박스만 빼고 나머지) | 88.8%(1건 43.8%로 급락) | 87.4% | 96.5ms |

YOLO 박스로 텍스트만 골라내려는 두 방식 모두 recall이 오히려 떨어짐(List-item 박스 커버리지
부족, 또는 표 bbox 과대추정이 인접 텍스트를 삼킴) — **가공 없는 whole_page가 가장 빠르고
정확함**. 순도(67.3%, 표/차트 텍스트 혼입)는 recall을 희생하지 않는 선에서 별도 고도화 과제로
분리.

## 리딩오더(다중 컬럼) 복원 필요성 분석 — 불필요로 결론

컬럼 클러스터링으로 "다중 컬럼처럼 보이는" 페이지가 다수 발견됐으나, 렌더링해 육안 확인한 결과
전부 나란히 배치된 표/차트였고(학술논문형 좌우 2단 본문은 두 문서 어디에도 없음), 실제 서술형
본문에서는 리딩오더 위반이 **0건** — 유일한 위반 2건은 법적고지 정의 목록(boilerplate) 내부
미세 구조였음. **MinerU 등 리딩오더 복원 모델 도입은 이 두 문서 유형에 불필요**하다고 판단, 대신
필요성을 자동 판별하는 라우터를 구현(아래).

## PDF 난이도 판별 라우터 (easy → 우리 파이프라인 / hard → MinerU 위임)

```
difficulty_score = w1·columns + w2·overlap + w3·rotation + w4·font_variance + w5·figure_density
```

YOLO로 표/차트 영역을 먼저 제외한 뒤 남은 텍스트만 컬럼 클러스터링(columns/overlap, 가중치
0.30/0.25 — 유일하게 실측 검증된 신호라 최고 가중치) + 회전/폰트 변동계수/그림 밀도(각 0.15,
탐색적 신호) 가중합. **본문 분량 컬럼이 실제로 겹치면**(200자 이상씩) 가중합 점수와 무관하게
항상 hard로 확정(확증 신호를 가중합에 희석시키지 않도록 override) — 나머지는 가중합으로 보조 판단.

| 대상 | 판정 |
|---|---|
| 실제 문서 16페이지(LG CNS 6 + Construct 10) | 전부 **easy**(최고 점수 0.36, threshold 0.40 미만) |
| 합성 2단 텍스트 PDF(직접 제작, positive 검증용) | **hard**(override로 정확히 판정) |

## 텍스트 정제 4종 — 전부 골든셋 회귀 없이 적용

| 기법 | 내용 | 실측 근거 |
|---|---|---|
| **PUA 글리프 제거** | 불릿 아이콘 폰트가 표준 유니코드 없이 임베드돼 나오는 Private Use Area 코드포인트(U+E000~F8FF) 제거 | 26개 발생 위치 전수 확인(전부 불릿 앞 독립 토큰, 본문 손상 없음) |
| **헤더/푸터 제거** | 좌표(상하단 8% 여백) + 반복빈도(50%+ 페이지에서 반복, 페이지번호는 `#`로 치환해 템플릿 비교) | LG CNS `LG CNS [#]`/`# Research Center`, Construct `건설 Weekly #.#.#`/`하나증권•#` 정확히 탐지 |
| **Boilerplate(Compliance Notice) 탐지** | Sentence Embedding(BGE-M3) 기반 — SimHash/MinHash는 다른 증권사 간 패러프레이즈를 못 잡아 기각 | 정밀도 100%(진짜 본문 23개 오탐 0), 재현율 83%(10/12) |
| **구두점/기호/공백 정규화** | 줄임표(.../…), en/em-dash, ▲▼(방향 부호, 숫자 앞이면 +/-로 변환해 의미 보존), 원문자 숫자, 유니코드 마이너스(−), NBSP, 한글 자모분리(NFD→NFC) | 전부 실제 혼용 사례 확인 후 정규화, 음수 표기("-15%") 영향 없음 확인 |

**속도 최적화 2건(전부 판정 결과 100% 동일, 지연만 감소)**:

| 최적화 | 개선 | 상태 |
|---|---|---|
| 난이도 라우터+계층적 청킹의 YOLO 호출 통합(페이지당 1회로) | 문서 전체 기준 1.45배(Construct 10p: 2.735s→1.881s) | **적용 완료** — `yolo_layout.py` 공유 모듈 + `text_extraction.process_pdf()`로 배선(아래 "최종 파이프라인" 참고), 골든셋 recall 회귀 없음 확인 |
| Boilerplate 탐지 위치+Jaccard 사전 필터 | LG CNS 2.561s→0.231s(11.1배), Construct 1.243s→0.238s(5.2배) | 적용됨(`detect_boilerplate_paragraphs_fast`) |

## 청킹 3종 비교(계층적/시멘틱/문맥적) — 최종 채택: 문맥적(구조 기반 컨텍스트)

| 방식 | 핵심 아이디어 | 청킹 자체 지연 |
|---|---|---|
| **계층적(Hierarchical)** | YOLO 구조(Title/Section-header/Text/List-item) 경계를 그대로 청크 경계로, section_path를 메타데이터로 | 0.6~0.8s(LLM/임베딩 없음, 가장 저렴) |
| **시멘틱(Semantic)** | BGE-M3 문장 임베딩 유사도가 떨어지는 지점(주제 전환)을 경계로 | 임베딩 호출만(수 초) |
| **문맥적(Contextual)** | 각 청크 앞에 "이 청크가 문서의 어느 부분인지" 설명을 주입(Anthropic Contextual Retrieval) | 컨텍스트 생성 방식에 따라 다름(아래) |

3개 문서(LG CNS 애널리스트 본문, Construct 주간전략/뉴스)에서 청크를 하나씩 독립적으로 LLM에
줘서(실제 RAG 검색 상황 재현) 엔티티+핵심 사실 추출 → 골든셋(Claude 작성) 대비 recall 측정.
**셋 다 명확한 승자는 없었음**(대명사로 회사명을 지칭하는 애널리스트 서술문에서는 문맥적이
유리, 이미 엔티티를 반복 언급하는 뉴스형 콘텐츠에서는 이점 없음) — 사용자 결정으로 문맥적
청킹으로 고정.

### 문맥적 청킹 컨텍스트 생성 백엔드 A/B — 로컬 Qwen vs OpenAI vs Rule-based vs 최종 채택

| 백엔드 | entity_recall | fact_recall | 지연(평균) | 비고 |
|---|---|---|---|---|
| Qwen(청크마다 로컬 VLM 호출) | 89.3% | 61.7% | 101.1s | 정확하나 가장 느림 |
| OpenAI 경량 API(청크마다 호출) | 59.3% | 51.7% | 55.0s | 빠르지만 recall이 rule-based와 동일 — 유료 API 이득 없음, 폐기 |
| Rule-based, 정적 placeholder | 59.3% | 61.7% | 42.1s | 첫 시도 — 컨텍스트가 문서 제목 반복뿐이라 표현력 부족 |
| **Rule-based, 계층적 청킹의 실제 section_path(최종 채택)** | **96.0%** | **68.3%** | **57.3s** | **LLM 호출 0건으로 Qwen보다 recall도 높고 지연도 1.8배 낮음** |
| OpenAI(gpt-4o-mini), 계층적 청킹의 실제 section_path 기반 컨텍스트 | 96.0% | 68.3% | 72.7s | recall은 rule-based와 **완전히 동일**, API 호출 왕복 때문에 27% 더 느리고 비용 발생 — 이득 없음 |

**핵심 발견**: 문맥적 청킹의 진짜 가치는 "LLM이 청크마다 컨텍스트를 생성하느냐"가 아니라
"컨텍스트에 실제로 유용한 정보(회사명이 들어있는 섹션 경로)가 담기느냐"였음 — 계층적 청킹이
이미 계산해둔 청크별 section_path를 규칙 기반으로 그대로 조립하는 것만으로 LLM 기반 컨텍스트
생성(Qwen 또는 gpt-4o-mini)과 recall이 동일하거나 더 좋음, 비용은 0원. 일단 section_path로
필요한 정보가 청크에 들어가고 나면, 그걸 "규칙으로 조립하냐 LLM이 다시 문장으로 풀어쓰냐"는
하류 추출 단계 입장에서 차이가 없음(정보량 자체가 이미 충분해 LLM의 재표현이 부가가치를 안 만듦) —
SLM을 더 붙이는 시도는 이 사실을 재확인하는 것으로 끝남(사용자 제안으로 gpt-4o-mini 추가 검증).

## 최종 텍스트 파이프라인 — `text_extraction.process_pdf()`(YOLO 중복 호출 제거 적용 완료)

```
PDF 페이지
    │
    ▼
run_yolo_layout(model, page) — 페이지당 YOLO 호출은 여기 **딱 한 번**(yolo_layout.py 공유 모듈)
    │
    ├──▶ assess_page_difficulty(cached_boxes 재사용) ── hard ──▶ MinerU(외부 리더, 미연동)
    │         │ easy
    │         ▼
    ├──▶ whole_page(PyMuPDF get_text) 추출 — Recall 100%, 9.5ms
    │         │
    │         ▼
    │    정제: PUA 제거 + 헤더/푸터 제거 + 구두점/기호/공백 정규화(NFC)
    │
    └──▶ chunk_hierarchical(cached_boxes 재사용, 문단 경계 + section_path 계산)
              │
              ▼
         컨텍스트 주입: generate_context_rulebased(청크별 실제 section_path, LLM 호출 없음)
              │
              ▼
         (선택) Boilerplate 제거 — 문단 단위, Sentence Embedding, 위치+Jaccard 사전필터
              │
              ▼
         엔티티/사실 추출(Qwen2.5-VL) → 임베딩/검색 단계로 전달
```

**YOLO 중복 호출 문제 해결됨**: 이전엔 Reading Order Router와 계층적 청킹이 페이지당 YOLO를
각자 따로 호출했음(사용자가 이 비효율을 직접 지적) — `yolo_layout.run_yolo_layout()` 공유 모듈로
분리하고 `process_pdf()`가 페이지당 한 번만 호출해 양쪽에 `cached_boxes`로 넘기도록 배선 완료.
실측(Construct 10페이지 전체): **2.735s → 1.881s(1.45배)**, 판정/청크 결과는 완전 동일(YOLO
호출 자체가 결정적이라 같은 이미지에 같은 모델을 다시 돌리면 같은 결과가 나옴 — 골든셋 recall도
재검증해서 회귀 없음 확인). [5]에서 측정한 "1.342s→0.177s(7.6배)"는 YOLO 호출 자체만 격리한
수치라 더 컸고, 문서 전체 기준(텍스트 추출/정제/청킹 등 YOLO와 무관한 비용까지 포함)으로는
1.45배가 정확한 실측치.

전체 파이프라인 중 **LLM 호출은 마지막 엔티티/사실 추출 단계에서만 발생**(청킹·컨텍스트 생성
자체는 YOLO+규칙 기반이라 무료, gpt-4o-mini를 추가로 붙여봐도 이득이 없음을 확인) — 텍스트
추출 정확도는 LG CNS 100%(22/22), Construct 100%(정규화 표기 차이 3건 제외, 44/47)로 재확인 완료.

## 임베딩 모델 채택 — BGE-M3 vs dragonkue/BGE-m3-ko vs OpenAI text-embedding-3-small

청킹 다음 단계(인덱싱)로 넘어가기 전, Claude가 직접 만든 검색 정답셋(자연어 질의 15개, 실제
최종 파이프라인 청크 21개가 코퍼스)으로 표준 IR 평가(Recall@1/@3, MRR) 실시.

| 모델 | Recall@1 | Recall@3 | MRR | 코퍼스 임베딩 |
|---|---|---|---|---|
| BGE-M3(다국어 베이스) | 87% | 93% | 0.907 | 0.855s |
| **dragonkue/BGE-m3-ko(채택)** | 87% | **100%** | **0.922** | **0.738s** |
| OpenAI text-embedding-3-small | 87% | 93% | 0.911 | 1.305s |

Recall@1은 동률이지만 BGE-m3-ko만 Recall@3 100%(정답이 항상 top-3 안) + 최고 MRR + 최속 —
세 지표 전부 우세/동률이라 트레이드오프 없이 채택(`embedding.py`).

## 참고: 다루지 않은 것 / 다음 단계

- MinerU 연동 자체(외부 리더 호출)는 이번 범위 밖 — 라우터가 `route_to_mineru`/`hard_page_numbers`만 반환, 실제 hard 사례는 합성 PDF로만 검증(실제 문서에 아직 없음)
- page_classification 모듈 자체(has_text/has_table/has_image 판정)와의 YOLO 결과 공유는 아직 — 지금 통합한 건 text_processing **내부**(난이도 라우터 + 계층적 청킹)만이고, page_classification까지 포함해 페이지당 YOLO를 진짜 1번으로 묶으려면 그 모듈과의 배선이 추가로 필요(다음 단계)
- **벡터DB 인덱싱 + BM25 하이브리드 검색은 아직 미구현** — 임베딩 모델은 정했으나(BGE-m3-ko), 실제로 chromadb 등에 저장하고 검색하는 단계는 다음 작업
