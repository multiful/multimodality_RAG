# 2-3. Table Pipeline

## 측정에 사용한 데이터셋

`20260721_company_279243000.pdf` (교보증권 LG CNS(064400) 기업분석 리포트, 6페이지, 표 14개)

## 행(row) 지표에 대한 정의

아래 표에서 "N행 추출"은 **텍스트 줄 수가 아니라, 각 방식이 표를 구조화(행×열의 레코드)한 후
인식해낸 행의 개수**다. 예를 들어 pdfplumber `extract_table()`은 표 격자(테두리선/셀 경계)를
인식해야만 행을 만들어내므로, 페이지에 텍스트 자체는 있어도 격자 인식에 실패하면 그 표는 **0행**으로
집계된다 — "텍스트가 없다"가 아니라 "표로 구조화하지 못했다"는 뜻. 이 지표가 곧 "표에서 실제로
써먹을 수 있는 정보량"의 근사치라 전 실험에서 핵심 성능 지표로 사용했다.

## 1차 고도화 모델 (표 구조화 성능 비교)

**Baseline**: PyMuPDF로 표 위치 후보를 잡고 pdfplumber `extract_table()`로 내용 추출.

| 방식 | 핵심 아이디어 | 총 추출 행수 | 지연 | 채택 여부 |
|---|---|---|---|---|
| **Baseline**(pdfplumber + PyMuPDF) | 표 위치에 pdfplumber `extract_table()`만 적용 | **30행**(14개 표 중 0행 표 4개) | 거의 무료 | 출발점 |
| Docling Only(1-Stage) | YOLO 없이 PDF 페이지 전체를 Docling에 직접 입력 | 165행 | 18.9초(3.14초/page) | 보류(YOLO 병행보다 낮은 품질, 느림) |
| Docling, YOLO 크롭별 개별 파싱(순차) | YOLOv11n으로 표 위치 탐지 → 크롭 → Docling(TableFormer)으로 구조화 | **240행**(baseline 대비 **+700%**) | 27.6초(순차) | 품질 기준선으로 채택, 지연은 아래에서 추가 개선 |
| Docling 배치 처리(페이지당 크롭을 세로로 합쳐 1회 호출) | 호출 횟수를 14→6으로 줄여 지연 단축 시도 | 156행(**-35%**) | 18.9초(-31.6%) | **기각**(표 경계 혼동으로 품질 손실이 지연 절감보다 큼) |
| **Docling 프로세스 병렬**(동적 워커+웜풀) | 워커별 모델 1회 로딩 후 재사용, 워커 수 = min(표 개수, CPU, 8) | 240행(불변) | **10.65초(-61.4%)** | **채택**(품질 손실 0, 지연 대폭 절감) |
| 표 크기 기반 라우팅(작은 표→pdfplumber, 큰 표→Docling) | 표 높이 550px 미만은 저비용 처리로 건너뜀 | 192행(**-20%**) | 9.42초(-11.5%) | **기각**(품질 손실이 지연 절감보다 큼 — "작은 표=쉬운 표"라는 가정이 실제로는 틀렸음, 후속 라우팅 설계에 중요한 반례) |
| YOLOv26n으로 교체(vs YOLOv11n) | 더 빠르다고 알려진 최신 YOLO로 표 위치 탐지 자체를 교체 | 149행(**-38%**) | 유사 | **기각**(page4류 밀집 표를 더 잘게/부정확하게 쪼갬 — YOLOv11n 유지) |
| Unstructured(Hi-Res, PaddleOCR 대체) | poppler/tesseract 없이 PyMuPDF+PaddleOCR로 우회 구현 | 페이지 분류는 최고(6/6) | **23.7초/page**(page4는 69초) | **탈락**(속도 최악, 네이티브 의존성 부담) |

- 이후에도 크롭 패딩/DPI 보정, 워커 수 추가 증대 등 여러 최적화를 시도했으나 전부 품질 저하 또는 지연 악화로 기각(상세: `실험.md` [12], [15])

## 표 복잡도 기반 라우팅 (Adaptive Table Complexity Router)

- 표 크기가 아니라 "pdfplumber가 실제로 그 표를 읽어냈는가"(fill_ratio, 컬럼 일관성, 빈 셀 비율)를 신호로 SIMPLE/COMPLEX 분류
- SIMPLE 표는 pdfplumber만으로 처리(Docling 호출 생략), COMPLEX 표만 Docling으로 처리
- 크기 기반 라우팅(작은 표=쉬운 표 가정)은 실패했던 반면, 이 방식은 품질 손실 없이 안전하게 동작함을 검증

## 진짜 병목은 표 추출이 아니라 엔티티 추출이었다

실측 결과, 표 구조화 단계는 전체 파이프라인 지연의 **1~3%**에 불과했고, 진짜 병목은 **엔티티 추출(LLM) 단계가 전체의 54~63%**를 차지하고 있었다(재무제표 6개가 밀집된 페이지 하나의 프롬프트가 거대해져 500초 이상 소요).

**개선**: 표를 재무제표(Finance) / 계약·일반(Contract·General)으로 분류해, 재무제표는 회사명이 거의 등장하지 않는다는 특성을 이용해 LLM 호출 없이 규칙 기반(정규식)으로 처리하고, 계약·일반 표만 LLM에 전달.

| 지표 | 개선 전 | 개선 후 |
|---|---|---|
| 총 처리시간 | 1097.5초 | **462.8초(-57.8%)** |
| Recall | 90% | 90%(동일 유지) |
| Precision | 81.2% | 94.4% |

## Trouble Shooting: Recall 9/10 문제 — "우리은행" 미탐지

### 문제 정의
표 파이프라인의 모든 개선 버전에서 10개 타겟 엔티티 중 **"우리은행" 1개만 지속적으로 누락**(Recall 90% 고정). 같은 표(도표4 수주공시)의 다른 계약상대(LG전자, 네이버클라우드, NH농협은행)는 전부 정상 탐지됨.

| 계약일 | 계약명 | 계약 상대 |
|---|---|---|
| 2026-04-01 | 고양 삼송 데이터 센터 위탁운영 | **주식회사 우리은행 외 1** |
| 2026-02-12 | 고양 삼송 IT플랫폼 센터 2단계 구축공사 | **주식회사 우리은행 외 1** |

### 원인 분석

1. **PDF 텍스트 추출 단계의 글자 간격 손상**: 해당 페이지는 pdfplumber로 추출 시 한글이 음절 단위로 띄어써짐(`주 식 회 사 우 리 은 행 외 1`). 표 전체를 한 덩어리로 읽는 방식에서는 이 손상이 그대로 LLM에 노출됨
2. **"외 1" 접미사가 엔티티명에 직접 결합**: 다른 계약상대는 단독으로 깔끔하게 등장하는 반면, "우리은행"만 "외 1"(그 외 1개사)이라는 부가 설명이 바로 뒤에 붙어 있어 모델이 독립된 고유명사로 분리하기 더 어려웠던 것으로 추정
3. **(부가 요인) Docling OCR 품질 저하**: 같은 표를 Docling으로 파싱해도 이 영역의 OCR 정확도가 낮아 표 구조화 단계에서도 이중으로 손실 발생

### TATR 고도화 과정: 도입 전 3단계 검증

"우리은행" 문제 해결책으로 TATR을 제안하기 전에, TATR이 정말 Docling보다 나은지 3단계로 실측 검증했다.

**주의(비교 기준 정정)**: TATR과 비교할 Docling 기준선은 1차 고도화에서 실제로 채택된 **프로세스
병렬(8워커) 버전**이어야 한다 — 아래 1단계 표의 "20.24초"는 이후 별도 실험(TableFormerMode/MPS
설정 효과만 순수하게 격리해 보려고 일부러 순차 처리로 돌린 것)에서 가져온 숫자라 채택 모델(10.65초
계열)보다 느리게 나온다. 그래서 2단계부터는 같은 12개 COMPLEX 표를 **8워커 프로세스 병렬로 돌린
Docling(10.927초)**을 기준선으로 다시 비교했다.

**1단계 — 구조 탐지만 비교 (공정하지 않은 1차 비교, 참고용)**

| 모델 | 12개 표 순차 처리 | 총 detected rows |
|---|---|---|
| Docling(TableFormer, 순차, OCR+구조+텍스트 전부) | 20.24초 | 228행 |
| TATR(구조 bbox만, 텍스트 없음) | 1.38초 | 242행 |

TATR은 row bbox만 뱉고 텍스트 추출은 안 하므로, 이 비교는 "완성된 표 데이터"끼리의 비교가 아니라 불공정하다.

**2단계 — TATR(구조) + pdfplumber(텍스트) 결합 vs 실제 채택된 Docling 프로세스 병렬(정정된 공정 비교)**

TATR이 찾은 row bbox를 pdfplumber에 넘겨 그 좌표의 실제 텍스트를 채워 넣는 방식으로, **1차 고도화에서
실제 채택된 8워커 프로세스 병렬 Docling**과 재비교:

| 방식 | 12개 표 처리 시간 | 최종 사용 가능 행수 |
|---|---|---|
| **Docling(TableFormer), 8워커 프로세스 병렬(1차 채택 모델)** | 10.927초 | 228행 |
| **TATR(구조) + pdfplumber(텍스트)** | **3.25초** | **228행 — 완전 동일** |

동일 품질에 **3.36배 빠름**(순차 대비 6.2배가 아니라, 실제 채택된 병렬 버전 대비 3.36배가 정확한 수치).
표 단위로 보면 더 흥미로운 지점도 있었다: Docling이 OCR 실패로 0행이었던 표를 TATR+pdfplumber는 7행 복원했고, 반대로 YOLO가 차트를 표로 오탐한 영역에서는 Docling이 OCR을 강행해 의미 없는 가짜 행 11개를 만들어낸 반면 TATR+pdfplumber는 정직하게 0행을 반환 — 오탐 상황에 대한 강건성도 확인됐다.

**3단계 — Padding × DPI 3×3 그리드로 최적 크롭 조건 탐색**

TATR용 표 크롭을 (No padding / 균일 padding / Adaptive padding) × (150 / 200 / 300dpi) 9가지 조합으로 만들어 어떤 전처리가 최선인지 탐색:

| 조합 | usable rows(12개 표 합계) |
|---|---:|
| no_padding + 150dpi | 225 |
| no_padding + 300dpi | 230 |
| padding + 150dpi | 239 |
| padding + 300dpi | 234 |
| adaptive_padding + 150dpi | 240 |
| **adaptive_padding + 300dpi** | **242(최고)** |

**핵심 발견 — TATR은 Docling과 정반대로 패딩에서 이득을 본다**: 표 크롭 고도화 실험(패딩+DPI)을 Docling에 적용했을 때는 여백을 추가할수록 인접 요소가 노이즈로 섞여 오히려 품질이 나빠졌었다(240→225행). 반면 TATR(DETR 기반 구조)은 표 경계를 명확히 봐야 첫/마지막 행을 놓치지 않아, 패딩이 있을수록·비대칭(adaptive)일수록 꾸준히 좋아졌다(225→242행). **"패딩이 좋다/나쁘다"는 모델마다 다른 결론**이라는 게 이 실험의 핵심 교훈이다. DPI를 400/500/600까지 더 올려봐도 242→240→242→240으로 완전히 평평해, 300dpi가 진짜 최적점(플래토)임도 확인했다.

**추가 검증 — "표는 무조건 TATR 고정"은 위험**: 이미 pdfplumber 단독으로 잘 처리되던 단순한 표(SIMPLE) 2개에도 TATR을 적용해봤더니 오히려 후퇴했다(12행→11행, 2행→0행 — 아주 작은 표에서 TATR의 row 탐지 자체가 실패). 따라서 **표 복잡도 라우팅(SIMPLE/COMPLEX 분류)은 그대로 유지하고, COMPLEX로 분류된 표의 처리 엔진만 Docling에서 TATR(adaptive_padding+300dpi)로 교체**하는 것으로 최종 결론.

### 해결: TATR(Table Transformer) 기반 Row 단위 추출

표 전체를 한 번에 텍스트로 읽는 대신:
1. **TATR**로 표 내부 개별 row의 경계(bounding box)를 구조적으로 먼저 탐지
2. 탐지된 각 row 좌표를 pdfplumber에 넘겨 **행 단위로 독립적으로** 텍스트 추출

행 경계가 명확해지면서 "주식회사 우리은행 외 1"이 다른 행과 섞이지 않고 하나의 완결된 문장으로 분리되어 LLM에 전달되고, 모델이 이를 정확히 하나의 엔티티로 인식하게 됨.

### 결과

| 지표 | Docling 기반(기존) | **TATR 기반(개선 후)** |
|---|---|---|
| Recall | 90%(9/10) — 우리은행 누락 | **100%(10/10)** |
| F1 | 92.2% | **96.6%** |
| 표 단계 소요 | 12.16초 | **4.11초(3배 빠름)** |

문서 앵커 자동화, 표 단위 분리 처리, 표-타입별 라우팅 등 여러 시도 중 어느 것도 이 문제를 해결하지 못했고, **표 구조 인식을 row 단위로 세밀화**하는 것이 근본 해결책이었다.

## 최종 표 처리 파이프라인

```
Table Page
    │
    ▼
YOLOv11n (Table Detection)
    │
    ▼
Adaptive Complexity Router (fill_ratio 기반 SIMPLE/COMPLEX 판정)
    │
    ├─ SIMPLE  → pdfplumber 단독
    │
    └─ COMPLEX → TATR(adaptive_padding+300dpi) + pdfplumber 텍스트 매핑
                     │
                     ▼
            Table Type Classification
                     │
            ┌────────┴────────┐
        Finance            Contract/General
            │                    │
      Rule Extractor          LLM
            │                    │
            └────────┬───────────┘
                      ▼
          Entity Extraction (Qwen2.5-VL,
          repetition_penalty=1.3, Grounding Filter)
```

**참고**: TATR+pdfplumber 방식은 텍스트 레이어가 있는 디지털 PDF에서만 유효하며, 스캔본에는 Docling의 내장 OCR로 폴백이 필요함(미검증).
