# 2-2. Page Classification Pipeline

- 측정에 사용한 데이터셋

`20260721_company_279243000.pdf` (교보증권 LG CNS(064400) 기업분석 리포트, 6페이지)

- **범위**: 페이지가 텍스트/표/이미지 중 무엇을 포함하는지 판별하고, 표는 위치(bbox)까지 찾아 크롭한다. 표/이미지 **내용**을 실제로 해석·추출하는 것은 이 범주 밖(→ Table Pipeline로 이관)
- **평가 우선순위**: Recall > Precision, 지연은 감소하면서 성능 저하가 크지 않으면 채택
- **정답 데이터**: `ground_truth_pages.json`(페이지별 실제 has_text/has_table/has_image 라벨)

## baseline : pdfplumber + PyMuPDF ("PyMuPDF Fast Scan")

- **방식**: pdfplumber `find_tables()`(표 감지) + PyMuPDF 래스터 이미지 수/벡터 드로잉 수(이미지 감지) 결합. LLM 미사용, 메타데이터만 검사
- **성능**:

| 라벨 | Accuracy | Precision | Recall | F1 |
|---|---|---|---|---|
| has_text | 100.0% | 100.0% | 100.0% | 100.0% |
| has_table | 83.3% | 100.0% | 83.3% | 90.9% |
| has_image | 66.7% | 66.7% | 100.0% | 80.0% |

- 전체 라벨 정확도 83.3%(18개 중), 페이지 완전일치 3/6, 지연 평균 161.8ms/page(매우 빠름)
- **오류**: has_table 미탐(page3, 테두리선 약함) / has_image 오탐(page4·6, 표 격자선을 벡터 드로잉으로 오인)

## 1차 고도화 모델

| 방식 | 핵심 아이디어 | 정확도(전 라벨) | 처리 시간 |
| --- | --- | --- | --- |
| **Baseline(pdfplumber+PyMuPDF)** | find_tables()+래스터/벡터 카운트, LLM 미사용 | 83.3%(has_table 83.3%, has_image 66.7%) | 161.8ms/page |
| **YOLOv11n Crop(채택)** | `Armaggheddon/yolo11-document-layout`(DocLayNet 11클래스)로 레이아웃 감지, 'Table'/'Picture' 클래스 판정 + bbox 크롭 저장 | **100%(전 라벨)**, 페이지 완전일치 **6/6** | 212.2ms/page |
| **YOLOv26n Crop(비교)** | YOLOv11n과 동일 로직, 모델만 YOLOv26n(2026-01-14 출시, 공식 43% 빠른 CPU 추론 주장)으로 교체 | 100%(동률) | 219.4ms/page(체감 차이 없음) |

- YOLOv11n 도입으로 baseline의 두 오류(표 미탐, 이미지 오탐) 전부 해결
- YOLOv26n은 페이지 분류 정확도·지연 모두 YOLOv11n과 동률이지만 **표 crop 개수가 14개→13개로 줄어듦**(표 1개를 못 찾음) — 이 차이가 다음 단계(표 내용 추출)에 미치는 영향은 이 범주 밖이라 Table Pipeline에서 별도 검증(→ Trouble shooting 참고)

## 1차 고도화 ++ 추가 개선 시도: 표 크롭 고도화

YOLOv26n의 표 crop 손실을 만회하기 위해, 크롭 자체를 정교화하면 개선되는지 검증:

1. **Adaptive Padding**: 150dpi 기준 상단 +35px(캡션 자연 포함) / 좌우·하단 +12px(타이트, 인접 표 침범 방지)
2. **Spatial Nearest Caption Mapping**: 표 bbox와 'Caption'/'Section-header' bbox 간 유클리드 거리로 최근접 캡션 결합(200px 초과 시 매칭 포기)
3. **DPI Sub-rendering**: YOLO 탐지는 150dpi 그대로, 최종 크롭만 PyMuPDF `clip`으로 300dpi 부분 재렌더링

| 모델 | 표 개수 | 캡션 매칭(200px 이내) | 매칭 안 됨 |
| --- | --- | --- | --- |
| YOLOv11n | 14 | 3개(21%) | 11개 |
| YOLOv26n | 13 | 4개(31%) | 9개 |

- page4처럼 표가 밀집된 페이지는 표마다 전용 캡션이 없는 경우가 많아(공용 섹션 헤더 1개를 여러 표가 공유) 캡션 매칭 커버리지 자체는 낮음(21~31%)
- **표 내용 추출 품질에 대한 실제 효과는 Table Pipeline에서 측정** — 결과: YOLOv11n은 품질·지연 둘 다 악화, YOLOv26n은 소폭 개선하나 여전히 YOLOv11n에 못 미침 → **기각**, page_classification 단계의 크롭 방식은 변경하지 않음(아래 Trouble shooting 참고)

## Trouble shooting

- **문제**: 표 크롭 고도화(패딩+캡션매핑+DPI 재렌더링)를 도입했더니, 기대와 달리 **YOLOv11n의 표 내용 추출 품질이 오히려 떨어짐**(240→225행, -6.2%)이고 지연도 대폭 증가(+140%). YOLOv26n은 소폭 개선(149→164행)했지만 여전히 YOLOv11n에 못 미침 — "왜 이런 정반대 결과가 나오는지" 원인이 불명확했음

- **원인 분석 — 대조군으로 변수 분리**: "패딩"과 "DPI 재렌더링" 중 무엇이 실제 원인인지 알 수 없어, 패딩만 단독 적용(150dpi 유지, 캡션매핑 없음)한 대조군을 별도로 만들어 재측정

| 모델 | 일반 크롭 | **패딩만(150dpi 유지)** | 패딩+캡션+DPI300(결합) |
| --- | --- | --- | --- |
| YOLOv11n | 240행, 10.65초 | **220행(-8.3%), 지연 거의 그대로** | 225행(-6.2%), 지연 +140% |
| YOLOv26n | 149행, 5.638초 | **148행(사실상 불변), 지연 거의 그대로** | 164행(+10.1%), 지연 +10.3% |

- **결론이 명확해짐**:
    - **YOLOv11n의 품질 저하는 패딩 자체가 원인**(패딩만으로도 240→220행 하락) — 이미 적절히 타이트한 bbox에 여백을 더하면 인접 표/여백이 섞여 표 구조 인식에 노이즈가 됨. DPI를 더해도 220→225로 거의 안 바뀜(오차 수준)
    - **YOLOv26n의 소폭 개선은 패딩이 아니라 DPI 재렌더링이 원인**(패딩만으로는 149→148, 사실상 무변화) — YOLOv26n은 bbox 자체가 더 작아(타이트) 150dpi로는 픽셀 수가 부족한데, 300dpi로 재렌더링하면 같은 영역이 더 선명해져 구조 인식이 개선됨
    - 정리하면 **두 모델에 필요한 게 정반대**다 — YOLOv11n은 패딩이 해롭고(노이즈), YOLOv26n은 DPI가 이롭다(해상도 부족 보완). 하나의 결합 레시피로 둘 다 좋게 만들 수 없음
    - **지연 폭증(+140%)의 원인은 패딩이 아니라 DPI 300 업샘플링**(패딩만일 때는 지연 거의 그대로)

- **최종 결론**: YOLOv26n의 DPI 이득(+10%)도 YOLOv11n의 절대 성능(240행)엔 못 미치므로, **기존 채택(YOLOv11n, 가공 없는 플레인 크롭) 유지가 최선** — page_classification 단계의 개선은 여기서 마무리
