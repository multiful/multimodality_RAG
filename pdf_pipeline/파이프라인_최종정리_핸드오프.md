## 0. 페이지 분류 기법 + 페이지 난이도 분류 기법

**모델/기법**:
- **레이아웃 탐지**: `YOLOv11n`(ultralytics, 커스텀 학습 `yolo11n_doc_layout.pt`) — 페이지를 150dpi로 렌더링해 텍스트/표/이미지 bbox를 한 번에 탐지. `pdf_pipeline/yolo_layout.py`의 `run_yolo_layout()`이 페이지당 정확히 **1회**만 호출되고, 그 결과(`page_boxes`)를 page_classification/text_processing/table_processing 세 곳이 전부 공유·재사용(중복 추론 제거, [8]/[10]/[26] 최적화 이미 적용됨).
- **1차 신속검사**: `PyMuPDF`(fitz) 메타데이터로 쉬운/어려운 PDF를 먼저 가른다(규칙 기반, 모델 없음).
- **난이도(easy/hard) 판정**: `pdf_pipeline/text_processing/reading_order_router.py`의 `assess_page_difficulty()` — 순수 규칙 기반. 컬럼이 기하학적으로 겹쳐도 원문 추출 순서가 컬럼별로 잘 그룹돼 있으면 문제 없다는 관찰을 반영해 `material_overlaps AND interleaving_excess>=2`를 **동시에** 만족할 때만 hard로 판정(Claude가 실제 문서 전수 검토해 캘리브레이션).
- **스캔본 감지**: `scanned_page_router.py`의 `detect_scanned_pages()` — 텍스트 레이어 유무 + 이미지가 페이지를 덮는 비율, 순수 PyMuPDF(모델 없음). hard 판정과는 독립적인 별개 신호.
- hard 페이지는 `MinerU`(PDF-Extract-Kit-1.0, pipeline 백엔드)로 라우팅.

**Construct 실행 결과**: 10페이지 분류 0.76s(YOLO 로드 포함), 스캔본 0개, hard 페이지 0개(전 페이지 easy — 우리 파이프라인이 끝까지 처리).

## 1. 이미지 최종 파이프라인 + 구조화 출력 구조

**경로**: `pdf_pipeline/image_processing/` (s1_parse.py → s2_onestop_mineru.py). 오늘 `~/minerU`(별도 Python 3.11 venv)로 **실제 실행**해 검증(이전엔 이 환경에 MinerU가 없어 대체 카드로만 검증했었음 — 이번에 실제 MinerU 연결까지 마침).

**모델 스택(단계별)**:
| 단계 | 모델 | 비고 |
|---|---|---|
| [0] 레이아웃+OCR | `MinerU`(PDF-Extract-Kit-1.0, pipeline 백엔드) | chart/image/table bbox + 캡션/각주 OCR |
| [1] 규칙필터 | 없음(크기/종횡비 규칙) | chart는 크기 무관 통과 |
| [2] 크롭 OCR | `PP-OCRv6`(det) + `PP-OCRv5`(rec, korean) | MinerU 내장, ~0.4~0.5s/장 |
| [3] 그림 분류기 | `DocumentFigureClassifier-v2.5`(EfficientNet-B0, 4M) | 26종 세부라벨, MinerU 탐지와 상호검증만(판정 자체는 안 바꿈) |
| [4a] 차트→표 (옵션) | `MinerU2.5-Pro-2605-1.2B` | 15.5s/장, 이번 실행에선 미사용(OCR만으로 충분해 생략, 근거는 §4) |
| [4b] 서술형 해석 (옵션) | `qwen3:8b`(Ollama, 로컬) | 4a의 표를 문체 변환, 3.5s/장 |
| [5] 구조화 출력 (옵션) | `gpt-4o-mini` | 카드별 정성 메타데이터 |

**Construct 실행 결과**: MinerU 파싱 34.0s(10p) → OCR/분류 12.0s → 총 32블록(chart/image 16 useful, table 16건은 판정 없이 표 브랜치로 인계`status=handoff`). `--with-chart-analysis` 없이도 PP-OCR만으로 "도표 3. 건설업종 종목 주간 수익률" 차트에서 회사명+수치가 정확히 읽혔음(§4 참고) — 이번 문서는 4a/4b 없이도 충분했다.

**구조화 출력 스키마**(`structured_output.ImageMetadata`, `gpt-4o-mini`): 실제 예시(도표 1. KOSPI 및 건설섹터 수익률 카드) —
```json
{
  "image_type": "chart",
  "caption_or_title": "도표 1. KOSPI 및 건설섹터 수익률",
  "entities_mentioned": ["Quantiwise", "하나증권"],
  "described_content": "이 차트는 KOSPI와 건설섹터의 수익률 변화를 보여준다.",
  "key_values_or_trend": "KOSPI가 400, 건설업이 360 수준에서 변화하고 있다.",
  "time_period": "25.1부터 26.7까지"

```
안정 필드(`onestop_cards.jsonl`): `image_id, doc_id, page, block_type, bbox, det_score, caption, footnote, status, filter_stage, embed_text, crop` — 항상 채워짐. `ocr{text,n_boxes,mean_conf}`/`clf_label`/`chart_table`/`narrative`는 옵션 단계를 켰을 때만.

**⚠ 이번에 발견·수정한 이슈 3건**:
1. **경로 불일치(버그)**: `image_processing/common.py`의 `PROJECT_ROOT`가 리포 루트가 아니라 `pdf_pipeline/` 자신이라(`Path(__file__).resolve().parent.parent`), 실제 산출물은 `pdf_pipeline/data/onestop/{doc_id}/`에 쓰이는데 데모(`run_investment_opinion_demo.py`)는 리포 루트 기준 `data/onestop/`을 보고 있어서 **실제 MinerU를 돌려도 항상 폴백 카드만 쓰던** 상태였음. `ONESTOP_CARDS_PATH`를 `pdf_pipeline/data/onestop/`로 수정.
2. **부호 오독(생성 단계 버그, §6에서 상술)**: 차트 OCR 텍스트의 "값(N)" 괄호 표기(한국 증권 리포트 관례상 음수/하락)를 LLM이 잘못 읽는 문제 — 데이터 정규화로 해결.
3. **캐싱 미배선**: `common.py`에 content-hash 기반 로컬 디스크 캐시(`cache_get`/`cache_put`, `data/cache/{stage}/{hash}.json`)가 이미 구현돼 있었는데 `ocr_crop`/`chart_table_extract`/`narrative_from_table` 어디서도 호출을 안 해서 죽은 코드였음 — 세 함수 모두에 배선. **실측**(LGCNS, `--force`로 카드 단위 resume을 우회하고 content-hash 캐시만으로 검증): 1차 실행 7.4s(캐시 생성, "MinerU OCR 로드 완료" 로그 있음) → 2차 실행 **4.1s**(OCR 엔진 로드 로그 자체가 사라짐 — 6개 크롭 전부 캐시 히트). Redis 아님, 로컬 파일 캐시라 별도 인프라/키 불필요. 실패(None) 결과는 캐시하지 않음(일시 오류 영구화 방지).

### "기업명 및 티커" DB 검색(ERD 하단 노드) — 연결 여부 점검

원본 ERD(`pdf_pipeline/reference/ERD.png`)를 보면 Entity Fusion → DB(supabase) 아래에 "기업명 및 티커"(재무제표 핵심지표 요약을 옆에 붙여서) 노드가 있고, 별도로 임시DB→Hybrid Search→Rank Fusion 결과도 같은 노드로 들어온다 — 즉 **두 개의 Supabase 조회(증거 검색 + 기업 식별)를 병렬로 하고 합치는 설계**다. 점검 결과:

- **실제 연결된 곳은 `streamlit/main.py`뿐이다.** `answer_question(ticker, query)`가 `financial_chunks`/`company_profile_chunks`(dense, `text-embedding-3-small`, 199개 KOSPI200 종목 데이터 있음, 실측 확인)와 `document_evidence`(hybrid, BGE-m3-ko) **세 테이블을 순차로**(병렬 아님) 조회해 합친다. 다만 여기서도 `ticker`는 **검색으로 알아내는 게 아니라 UI에서 사용자가 미리 골라서 주는 값**이다(홈 화면 종목 카드 클릭, 또는 업로드 시 드롭다운 선택) — "PDF 안 엔티티로 DB를 검색해서 티커를 자동으로 찾아낸다"는 흐름은 **어디에도 없다.**
- `run_investment_opinion_demo.py`(이 문서가 검증해온 pdf_pipeline 데모)는 `company_profile_chunks`/`financial_chunks`를 아예 안 건드린다 — `ticker`가 함수 인자로 고정 전달될 뿐.
- **사용자 지시대로 "dense로 추출"을 실제 테스트**: PDF에서 뽑힌 실제 엔티티명(GS건설/현대건설/IPARK현대산업개발/삼성물산)으로 `company_profile_chunks`(dense, GPTEmbedder)를 직접 질의해봤다:

| 질의(한글 기업명) | 정답 티커(yfinance 확인) | dense 검색 top-2 | 정답 히트? |
|---|---|---|---|
| GS건설 | 006360.KS | 078930.KS, 267270.KS | ❌ |
| 현대건설 | 000720.KS | 267270.KS, **000720.KS** | ✅(2위) |
| IPARK현대산업개발 | 294870.KS | 024110.KS, 000720.KS | ❌ |
| 삼성물산 | 028260.KS | 093370.KS, 062040.KS | ❌ |

**4건 중 1건만 부분적으로 맞음 — 이대로 자동 연결하면 안 됨.** 원인: `company_profile_chunks`는 한글이 아니라 **영문 회사명/사업설명**(yfinance longName 등)만 담고 있어서, 한글 기업명 질의와 임베딩 공간에서 제대로 안 붙는다. 이건 애초에 임베딩(의미 유사도)으로 풀 문제가 아니라 **정확한 이름→티커 조회(lookup) 문제** — dense search를 쓰는 게 도구 선택 자체가 안 맞다(프로젝트가 이미 여러 번 확인한 패턴: "이게 진짜 ML/임베딩이 필요한 문제인가"). `data_collection/kospi200_tickers.py`가 pykrx로 `{한글명, 티커}` 쌍을 가져올 수 있지만 이 환경엔 `pykrx` 미설치 + KRX 로그인 계정 필요라 지금 당장 못 씀.

**권장(구현은 보류, 사용자 확인 후 진행)**: 한글명→티커는 (a) `KOSPI200_output`에 이미 있는 199개 프로필을 영文名 그대로 놔두고 한글 별칭 컬럼만 추가하거나, (b) `pykrx`(KRX 계정 필요) 또는 (c) 이미 PDF 구조화 출력이 뽑아준 entities를 그대로 evidence에 태그로 남기고(지금도 됨, §1~3의 structured_metadata.entities) 별도 매칭 없이 LLM이 컨텍스트 안에서 처리하게 두는 현재 방식을 유지 — 셋 중 하나 확정 필요.

## 5. 텍스트 인덱싱 과정과 검색 파이프라인 요약

**경로**: `pdf_pipeline/text_processing/index_text.py` + `pdf_pipeline/entity_fusion.py`.

- **인덱싱**: `dragonkue/BGE-m3-ko`(dense, 1024차원, `SentenceTransformer`) + `BM25Okapi`(sparse, 한글 어절 토크나이저). 3브랜치가 각자 계산한 임베딩을 재사용해 통합 인덱스를 만들어 **재임베딩 비용 없음**.
- **Rank Fusion**: `weighted_sum`(min-max 정규화 후 가중합, dense 0.7/bm25 0.3) 또는 `rrf`(Reciprocal Rank Fusion, k=60) — 실측(15개 factoid 질의) 하이브리드가 dense-only 대비 recall@1 **86.7%→93.3%**.
- **쿼리 분류 라우팅**(`route_search`): `gpt-4o` 기반 이진 분류(keyword_specific / abstract) — 25개 라벨셋 실측 규칙기반 80% vs gpt-4o-mini 84% vs **gpt-4o 92%**(채택). **키워드형(keyword_specific)은 `hybrid_search(fusion="rrf")`(dense+BM25)로, 추상형(abstract)은 BM25를 뺀 dense 전용 경로(HyDE 또는 MQE, `use_bm25=False`)로** 갈라진다 — abstract 질의에서 BM25가 순수 노이즈이거나 오히려 해가 된다는 실측([43])이 이 분기의 근거. abstract는 다시 문서 엔티티 수로 한 번 더 갈라 단일기업(entity_count<=1)→HyDE(`gpt-4o-mini`), 다기업(>1)→MQE(`gpt-4o-mini`, 하위질의 4개 생성 후 RRF 융합).
- **entity_count 사전계산**(`precompute_entity_count`): 인제스트 시점에 1회만 계산해 캐시 — 하나증권 포맷이면 정규식(LLM 호출 0회), 아니면 gpt-4o-mini 폴백.
- **Supabase 재조회 경로 확인**: `entity_fusion.load_evidence_from_db(DB_URL, pdf_id="Construct")`로 별도 프로세스에서 DB만 읽어 재인덱싱 → 검색 재실행까지 정상 동작 확인(재수집/재파싱 없이 "저장된 문서에 다시 질의" 가능).

**⚠ 캐싱 추가**: `load_evidence_from_db()`가 매 호출마다 DB 재조회+임베딩 문자열 파싱+BM25 전체 재구축을 하던 문제(사용자 지적) — `(pdf_id, ticker)` 키의 **프로세스 메모리 캐시**(TTL 60초, `invalidate_evidence_cache()`로 수동 무효화 가능)를 추가. **실측**(Construct): 1차 1.0s → 2차(캐시 히트) **0.0000s** → `invalidate_evidence_cache(pdf_id="Construct")` 후 3차 0.74s(정상 재조회 확인). Redis 불필요 — TextIndex가 numpy 배열+BM25Okapi 객체를 들고 있어 Redis에 넣으려면 pickle 직렬화가 추가로 필요한데, 지금 배포가 단일 프로세스라(README: 배포 미구현) 그 복잡도가 아직 정당화 안 됨. [49] RQ 백그라운드 잡처럼 나중에 upsert가 들어오는 경로는 TTL 60초 안에서만 살짝 stale할 수 있음 — 알고 있는 호출측은 즉시 무효화 가능.

### 쿼리 라우팅 재검증 — 4개 문서(핀터멘털/납기/LGCNS/KWave)로 확대 실측

기존 골든셋(`golden_set_event_insight.json`, KWave/LGCNS/납기)에 **핀터멘털**(entity_count=0, 개별종목 투자의견이 아니라 제약/바이오 섹터 트렌드형 리포트라 기존 3개와 다른 유형)을 추가해 `evaluate_retrieval_ab.py`를 4개 문서로 재실행(쿼리: "이 PDF에 나오는 기업의 이벤트 요약과 투자 인사이트를 도출해줘", summary/abstract형).

| 문서 | entity_count | route_search 선택 | 실측 더 나은 쪽 | 일치 |
|---|---:|---|---|---|
| KWave | 6 | MQE | MQE(ndcg 0.889 vs HyDE 0.849) | ✅ |
| LGCNS | 1 | HyDE | HyDE(ndcg 0.967 vs MQE 0.734) | ✅ |
| 납기 | 2 | MQE | MQE(ndcg 0.871 vs HyDE 0.505) | ✅ |
| 핀터멘털 | 0 | HyDE | HyDE(ndcg 0.784 vs MQE 0.626) | ✅ |

**4/4 전부 일치** — entity_count 기반 HyDE/MQE 라우팅이 더 넓은 표본(기존 2~3개→4개, entity_count 0/1/2/6 스펙트럼)에서도 재확인됨.

4개 문서 macro-average(동일 가중):

| 방법 | ndcg | mrr | 지연 |
|---|---:|---:|---:|
| dense-only | 0.554 | 0.317 | 0.05s |
| dense+BM25(hybrid) | 0.428~0.470 | 0.229~0.292 | 0.01~0.02s |
| 라우팅(구, 항상 MQE) | 0.754 | 0.625 | 1.21s |
| **라우팅(신, 엔티티인식)** | **0.848(최고)** | **0.875(최고)** | 2.60s |

abstract형 질의에서 순수 hybrid(dense+BM25)가 dense-only보다 평균적으로 더 나쁨(0.428~0.470 vs 0.554, KWave는 ndcg 0.000까지 하락)도 재확인 — `route_search()`가 abstract 질의엔 hybrid를 안 쓰고 HyDE/MQE(BM25 제외)로 우회하는 설계가 맞다는 근거.

**✅ [수정 완료] route_search를 데모에 배선함**: `run_investment_opinion_demo.py`의 검색 단계(구 7번)가 `entity_fusion.weighted_hybrid_search()`(항상 고정 hybrid) 대신 `index_text.route_search()`(4문서 평균 최고 성능, ndcg 0.848)를 쓰도록 교체 완료. 인제스트 직후 `precompute_entity_count()`도 추가([46] 권장대로 질의 지연에 안 노출되게).

**🔴 그런데 교체 직후 실측하니 이 Construct 쿼리에서 route_search가 오히려 오답을 냄 — 정직하게 기록**: 같은 쿼리로 재실행하니 qtype=`abstract`, entity_count=7~8(구조화 출력 유무에 따라 근소 차이)로 분류돼 MQE(BM25 제외) 경로를 탔고, 최종 답이 **"주간 수익률이 가장 높은 기업은 한샘(11.7%)"**으로 나왔다 — **틀렸다**. 한샘은 도표4(건자재업종) 소속이고, 질의는 명시적으로 "건설업종"을 물었다(정답은 도표3의 IPARK현대산업개발 +1.3%, §6 이전 실측 참고). 근거 로그로 원인 확인:
```
도표4(건자재업종) image score=0.0706  ← 1위
도표3(건설업종)   image score=0.0640  ← 5위, 근소하지만 역전당함
```
`weighted_hybrid_search()`(dense+BM25)로는 도표3이 0.940으로 압도적 1위(BM25가 캡션 "건설업종"/"주간 수익률" 문자 그대로를 정확 매칭)였는데, `route_search()`의 abstract 경로는 `use_bm25=False`로 BM25를 아예 뺀다([43] "abstract 질의엔 BM25가 노이즈"라는 검증된 설계) — 이 쿼리는 그러나 순수 abstract가 아니라 **"인사이트 도출"(진짜 abstract) + "건설업종 종목 중 최고 수익률 추출"(키워드/비교형)이 합쳐진 복합 질의**였고, 뒤쪽 절의 정밀 키워드 매칭 필요성을 분류기(rule-based·LLM 둘 다 "인사이트"/"도출" 트리거 단어 때문에 abstract로만 분류함, 복합 여부는 못 봄)가 못 잡아서 BM25를 통째로 잃은 게 원인.

→ **이게 바로 지난 턴에 논의한 "문맥별 분할 라우팅"이 실제로 필요하다는 최초의 구체적 실패 사례다.** 이전엔 "이론적으론 맞지만 이번 케이스는 단일 hybrid pass로 충분했다"고 썼는데, route_search로 바꾸고 나니 그 "충분했다"는 사실 weighted_hybrid_search가 BM25를 안 버렸기 때문이었다는 게 드러남.

**✅ [수정 완료] 쿼리 디콤포지션 구현·배선함**: `index_text.decompose_query()`/`decompose_and_route_search()` 신규 추가. gpt-4o로 질의를 성격별 하위질의로 분해(이 질의는 실제로 "abstract: 인사이트 도출" + "keyword_specific: 건설업종 최고 수익률 기업 추출" 2개로 정확히 쪼개짐, 실측 확인) → 하위질의마다 각자 전략(keyword_specific=hybrid RRF, abstract=HyDE/MQE)으로 검색 → **라운드로빈으로 인터리빙**해 합침(첫 시도는 순위기반 RRF 합산이었는데, abstract 하위질의가 코퍼스 전반에 점수를 흩뿌리면서 keyword_specific의 1위 결과가 오히려 희석되는 걸 실측으로 발견 — 라운드로빈으로 교체해 각 하위질의의 1위가 최상위 슬롯에 무조건 들어가게 고침). 결과: 도표3(건설업종)이 다시 상위권(2위)으로 복귀. `run_investment_opinion_demo.py`의 검색 단계를 route_search에서 이걸로 재교체.

**✅ [수정 완료] "기업명 및 티커" 자동 연결 — `company_entity_linking.py` 신규**: dense 대신 `pykrx.stock.get_market_ticker_name()`(로그인 불필요, 199/199 성공)으로 KOSPI200 199종목의 정확한 한글명을 캐시(`KOSPI200_output/kospi200_korean_names.json`)해두고, PDF 원문 텍스트에 그 이름이 등장하는지 정확 매칭한다. **오탐 함정 발견·수정**: "GS"(GS지주)가 "GS건설" 안에서, "동서"(동서식품)가 "아이에스동서" 안에서 오탐되는 걸 실측으로 발견 — 3자 이하 이름은 앞뒤가 한글로 안 이어질 때만(공백/구두점) 인정하도록 고쳐 오탐 0으로 해결(진짜 매칭은 "한샘"처럼 유지). Construct 실행 결과 **7개 기업 정확 매칭**(현대건설/한샘/삼성E&A/삼성물산/대우건설/한일시멘트/DL이앤씨) → `financial_summaries`/`company_profile_chunks`에서 실제 재무 요약을 PK로 직접 조회(검색 아니라 정확 조회라 오류 여지 없음)해 최종 프롬프트에 결합. 문서 근거 검색과 **ThreadPoolExecutor로 병렬 실행**(사용자 지적 "DB를 두 번 병렬로 검색해" 반영).

**✅ [수정 완료] 최종 생성 모델 업그레이드**: `gpt-4o-mini` → `gpt-4.1`(컨텍스트 ~1M, 더 강한 추론).

**🔴 그런데도 최종 재실행 결과 "1위 기업" 답이 또 틀렸다 — 새로운 원인, 정직하게 기록**: 검색은 이제 정확히 도표3을 상위권에 올렸고, DB 매칭도 정확했는데, 최종 답변이 "가장 높은 수익률은 금호건(5.2)"라고 답함 — 이것도 틀렸다(정답은 여전히 IPARK현대산업개발 +1.3%). 원인은 이번엔 로직이 아니라 **원본 OCR 데이터 자체의 결함**: 도표3 원문 OCR 텍스트가 "코오롱글로벌 (5.9) 금호건5.2) HL D&I (2.7)"인데, "금호건설(5.2)"에서 "설("이 통째로 유실돼 "금호건5.2)"로 깨져 들어왔다(PP-OCR의 박스 병합 오류로 추정). 여는 괄호가 없어 내 부호 정규화 정규식(`\((\d+)\)`, 여닫는 괄호 쌍 필요)이 이 항목을 못 잡았고, 그래서 LLM이 이걸 괄호 없는 양수로 읽었다 — 회사명 자체도 "금호건"으로 잘못 나와 실재하지 않는 이름이다(그래서 `company_entity_linking`이 매칭 안 한 것도 오히려 정합적 — 매칭 안 됐다는 게 "이 이름은 우리가 아는 진짜 기업이 아니다"라는 신호였는데 생성 단계가 그 신호를 활용 안 함). **이건 로직 버그가 아니라 원본 이미지 OCR 품질 문제라 리랭킹/디콤포지션으로 못 잡는 새로운 종류의 실패 — §"남은 과제"에 기록.**

**⚠ 추가로 발견한 세부 이슈**: 납기에서 `route_search()`가 실제로 탄 MQE 경로(ndcg 0.693)가 독립적으로 부른 `MQE(BM25 제외)`(ndcg 0.871)보다 낮음. 원인 추정: [44]가 지연 단축을 위해 "분류+하위질의 생성"을 `_classify_and_expand()` 호출 1번으로 합쳤는데, 이게 `mqe_search()`가 따로 쓰는 전용 하위질의 프롬프트보다 이 문서에서 품질이 낮은 하위질의를 만든 것으로 보임 — 라우팅 방향(MQE가 맞다) 자체는 옳았지만 [44]의 지연 최적화가 이 케이스에선 품질을 일부 희생시켰을 가능성. 재검토 필요.

**⚠ 오늘 발견·수정한 동시성 버그**: 3브랜치(text/table/image)를 `ThreadPoolExecutor`로 동시 실행하니, 텍스트 브랜치의 `BGE-m3-ko` 임베딩과 표 브랜치의 `TATR`이 **둘 다 자동으로 Apple MPS에 올라가 같은 시점에 다른 스레드에서 GPU 커널을 동시 제출**하면서 TATR 결과가 예외 없이 조용히 틀어지는 걸 실측으로 확인(같은 표가 격리 실행 27개 canonical 매칭 → 동시 실행 시 2개로 저하). **TATR을 CPU로 고정**해 해결(MPS 이득이 원래 9%뿐이라 손해 작음, [37] 참고). → **동시 실행 도입 시 반드시 함께 점검해야 할 클래스의 문제**로 팀에 공유 필요(다른 로컬 모델을 추가로 동시 실행에 얹을 때도 같은 문제가 재발할 수 있음).

**연결 상태 최종 점검**: `entity_fusion.py`/`index_text.py`/`ingest_queue.py`는 이번 세션에서 로직을 수정하지 않았고, 표/이미지 브랜치의 새 출력이 기존 인터페이스(`from_text_chunks`/`from_table_records`/`from_image_cards`)를 그대로 타고 들어가는 것만 확인 — **연결 안 된 부분 없음.**

## 6. 최종 생성 LLM 평가

**모델**: `gpt-4o-mini`(생성) + `gpt-4o-mini`(citation-check 검증) — `pdf_pipeline/citation_check.py`.

**설계**: 답변 생성 → `find_unsupported_numbers()`로 답변에 등장한 3자리 이상 숫자가 컨텍스트에 실재하는지 확인 → 없으면 피드백과 함께 최대 2회 재생성. `_retryable_create()`(tenacity, 429/5xx/타임아웃만 재시도)로 API 레벨 장애도 보호(오늘 배선 안 돼있던 걸 수정 — §부록 참고).

**Construct 실행 실측**:
- 1회 생성으로 통과(`unsupported_numbers: []`), 총 6.3s.
- **인용 정확도**: 답변에 등장한 수치(14.3%, 13.2%, 7.6%, 1.3%, 11.6%, 0.6%p, 11.7% 등)가 전부 컨텍스트에 실재 — citation-check가 정상 작동.
- **⚠ 그러나 citation-check를 통과한 답변이 처음엔 틀렸음**: gpt-4o-mini가 "값(N)" 괄호(한국 증권 리포트 관례상 **음수**)를 그냥 큰 숫자로 읽어 **GS건설(-14.3%, 실제로는 가장 큰 하락)을 "가장 높은 수익률"**이라고 답한 오류를 실측으로 발견. citation-check는 숫자가 컨텍스트에 "존재하는지"만 보고 부호는 안 봐서 이 오류를 못 걸러냄(모듈 자체 문서화된 한계: "단위 오귀속... 못 잡는다").
  - **1차 시도(프롬프트에 "괄호=음수" 지침 추가)**: 실패 — gpt-4o-mini가 지침을 안정적으로 안 지킴(동일하게 GS건설을 오답으로 지목).
  - **2차 시도(데이터 정규화)**: `block_type="chart"` 카드의 `embed_text`에서 `(N)` → `-N`으로 **인제스트 시점에 결정적으로 치환**. → 정답(IPARK현대산업개발, +1.3%)으로 정정 확인.
  - **교훈**: LLM의 즉석 해석(프롬프트 지침)에 기대는 것보다, 애매한 표기 관례는 **데이터 단계에서 정규화**하는 쪽이 훨씬 안정적 — 이번 사례가 그 근거.

**정정 후 최종 답변(발췌)**:
> "...건설업종에서 주간 수익률이 가장 높은 기업은 **IPARK현대산업개발**로, 1.3% 상승세를 기록하였습니다... 반면, **GS건설**은 -14.3%로 가장 큰 하락률을 보였으며..."

부가로 건자재업종 차트(도표4)에서 "한샘 11.7%"까지 자발적으로 인용 — 여러 소스(text+image) 근거를 실제로 종합하는 것도 확인됨.

→ **route_search만 도표4를 1위로 올려 역전**(abstract 경로가 BM25를 버려 "건설업종" 문자 매칭을 잃음, §5와 일치). `weighted_hybrid`는 BM25를 유지해 **정확하면서 55배 빠름**(0.05s). 단 §5 4문서 평균에선 순수 abstract 질의에서 routing이 이기므로 **weighted_hybrid가 항상 우월한 건 아님** — 이 복합/키워드 포함 질의에선 우연히 최적. 어쨌든 **검색은 병목이 아니다**(최대 2.78s ≪ 생성).

**(B) 생성 모델** (검색은 decompose 고정, 동일 컨텍스트 10.8K자, 기업DB매칭 7건):

| 모델 | 지연 | attempts | IPARK 정답 |
|---|---:|---:|:--:|
| **gpt-4o-mini** | **6.58s** | 1 | ✅ |
| gpt-4.1(현 데모) | 14.74s | 1 | ✅ |

→ **gpt-4.1이 gpt-4o-mini 대비 2.2배 느린데 정확성 이득은 없음**(이 질의에선 둘 다 정답, 재시도 0). §6/§남은과제9의 "gpt-4.1 교체 후 재시도 잦아 48s"의 비용이 이 A/B에서 지연으로 확인됨.

**검색+생성 end-to-end**:

| 경로 | 지연 |
|---|---:|
| 현 데모(decompose + gpt-4.1) | 17.52s |
| opt_A(decompose + gpt-4o-mini) | **9.36s** (−47%) |
| opt_B(weighted_hybrid + gpt-4o-mini) | **6.63s** (−62%) |

### 7.3 병목 판정 + 지연 개선(구현 완료)

**병목 = 생성 단계**(6.6~14.7s, §6에선 재시도로 최대 48s). 검색(≤2.78s)·로드(0.65s)는 부차적. BGE 콜드로드 10.4s는 프로세스 1회성(질의당 아님).

**구현한 개선 2건**(안전·기본값 불변):
1. **생성 모델 파라미터화**: `main(..., gen_model="gpt-4.1")` 추가(기본값 유지 → 무회귀). 지연 민감 호출은 `gen_model="gpt-4o-mini"`로 **2.2배 단축**(정확성 동일, 위 A/B 근거). 데모 stage8이 `model=gen_model` 사용.
2. **프롬프트 하드닝(재시도 꼬리 감소)**: 생성 프롬프트에 "기업 DB 원본 숫자는 단위 변환·반올림 없이 자릿수 그대로 인용" 지침 추가. §남은과제9의 원인(gpt-4.1이 원단위 금액을 억/백만으로 재표기 → citation-check가 근거없는 숫자로 오인 → 불필요 재생성)을 데이터가 아니라 지침으로 선제 차단(모델 불변, 오직 재시도만 줄임 → 다운사이드 없음).

**권장(기본값 변경은 팀 확인 후)**: 이 질의에선 gpt-4o-mini가 동급 정확·2.2배 빠름. 다만 1개 질의 근거라 **여러 질의로 확대 검증 후** 기본 모델 하향을 결정할 것(§6가 gpt-4.1을 "강한 추론" 목적으로 올린 맥락 존중). 지금은 **레버(gen_model)만 제공**.

### 7.4 파이프라인 진행 중 발견한 병목/이슈 체크
- **생성 재시도**가 최대 병목(위). → gen_model 레버 + 프롬프트 하드닝으로 완화.
- **route_search 단독은 이 복합질의에서 오답**(도표4 1위) — 데모는 이미 decompose로 교체돼 있어 안전하나, route_search를 직접 쓰는 `run_cband_full_pipeline_test.py`는 이 함정에 노출됨(참고).
- **BGE 콜드로드 10.4s**: 서비스화 시 모델 프리로드(워밍업)로 질의 지연에서 분리 필요(현재 데모는 매 프로세스 1회 지불).
- 검색 자체는 병목 아님 — decompose의 gpt-4o 분해 호출(≈2s)이 대부분. 키워드 포함 질의는 `weighted_hybrid`(0.05s)로도 정확하니, **질의 성격별로 decompose 스킵 조건**을 두면 검색 지연을 더 줄일 여지(향후 과제, §남은과제).

## 8. 텍스트겹침 OCR 손상 수정 ✅ + 건자재 질의 A/B + 메타/엔티티 평가 + 검색 냉정분석 — 재일

**대상/질의**: Construct(`362851000`) 동일 PDF. 새 질의 = **"이 PDF 기업 인사이트 도출 + 건자재업종 종목 중 주간 수익률 최고 기업"**. 골든셋은 PDF 텍스트레이어(권위)를 직접 읽어 제작(`pdf_pipeline/final/ground_truth_construct.json`). 재파싱 없이 Supabase evidence 로드 → A/B(`final/work/construct_ocr_fix_ab.py`, `mqe_ab.py`, `construct_ab.py`).

### 8.1 골든셋 (PDF 직접 검증)
- **도표3 건설업종 1위 = IPARK현대산업개발 +1.3%**(유일 상승). 금호건설 실제 **-35.2%**(최저).
- **도표4 건자재업종 1위 = KCC글라스 +2.5%**. 한샘 실제 **+1.7%**.

### 8.2 ✅ [해결] "텍스트 겹침" 차트 OCR 손상 수정 (금호건설·한샘 둘 다)
차트 이미지에서 텍스트가 겹쳐 PP-OCR이 깨지는 문제 — 두 건이 실제로 stored image evidence에 있었음:
- `...코오롱글로벌 -5.9 **금호건5.2)** HL D&I -2.7` — "금호건설(35.2)"의 "설(3" 유실 → 회사명 손상 + 여는괄호 소실 → +5.2 오독.
- `...KCC글라스 2.5 ... **한샘 11.7** ...` — "1.7"에 '1' 중복 → 11.7(축 최대 +4 **초과=off-chart**) → "건자재 최고"로 오답.

**수정 3종**(`run_investment_opinion_demo._normalize_chart_card_signs(cards, pdf_path=)`에 배선, 데이터 단계 결정적 처리):
1. `(N)` 완전괄호 → `-N` (기존).
2. **`N)` 닫는괄호만(여는괄호 유실) → `-N`** — 금호건설 부호(하락) 복원 → "최고"로 못 뽑힘.
3. **텍스트레이어(권위) 대조** — 이미지 OCR 숫자가 PDF 원문 페이지 숫자집합에 없으면 손상값으로 보고 `[OCR손상]`으로 치환(수치 제거). 한샘 "11.7"은 원문에 없고(원문엔 1.7) → 제거되어 "최고" 근거로 못 쓰임. **핵심: 베이스라인의 강점(깨끗한 텍스트레이어)으로 파이프라인의 약점(이미지 OCR 손상)을 교정.**

**실측(Construct, gpt-4o-mini N=4)**: 건자재 최고 질의에서 **한샘 오답픽 corrupted 1/4 → fixed 0/4**(손상값 11.7이 제거돼 KCC글라스가 정답으로 선택됨). ✅
**주의(X 잔여 리스크)**: 수정(3)은 PP-OCR과 PyMuPDF 반올림 표기가 다르면 정상값도 제거될 위험 — 이 문서군엔 깨끗했으나 여러 문서 오탐률 측정 필요(남은과제 1).
